S2. **`isClientOnly` misclassifies `--` before subcommand as client-only** — `pkg/kuberlr/resolver.go:270-322` [Gemini 3.1 Pro]

```go
for i := 0; i < len(args); i++ {
    arg := args[i]
    if arg == "--" {
        break
    }
    ...
}
if subcommand == "" {
    return true
}
```

`kubectl -- get pods` is syntactically valid: pflag treats `--` as the end-of-flags marker and Cobra reads the first positional as the subcommand. `isClientOnly` walks `args`, breaks on `--` at line 278-280, finds no subcommand, and returns `true` via the empty-`subcommand` fallback at line 312-314. The resolver then skips the probe and runs the embedded kubectl — exactly the misclassification the comment at lines 264-269 warns against. Probability is low (no documented `kubectl` recipe puts `--` before the subcommand) but the fallback's bias should still be "probe, don't skip."

Fix: take the next positional after `--` as the subcommand candidate before breaking.

```diff
 for i := 0; i < len(args); i++ {
     arg := args[i]
     if arg == "--" {
+        if subcommand == "" && i+1 < len(args) {
+            subcommand = args[i+1]
+        }
         break
     }
```

A `TestIsClientOnly` row covering `["--", "get", "pods"]` (expect `false`) would lock the fix in.

S3. **`streamGet`'s 256 MiB cap surfaces as "checksum mismatch", not as "binary too large"** — `pkg/kuberlr/downloader.go:105,128-130,169-184` [Claude Opus 4.7]

```go
if err := streamGet(ctx, base, io.MultiWriter(tmp, h), maxKubectlBytes); err != nil {
    tmp.Close()
    return fmt.Errorf("downloading %s: %w", base, err)
}
...
got := hex.EncodeToString(h.Sum(nil))
if got != want {
    return fmt.Errorf("checksum mismatch for %s: want %s, got %s", base, want, got)
}
```

If kubectl ever exceeds `maxKubectlBytes` (256 MiB), `io.LimitReader` returns silently after `maxBytes`, the digest covers a truncated body, and the caller sees a `checksum mismatch` error pointing at integrity instead of the size cap. `fetchSha512` already gets this right: a misconfigured mirror serving the binary at the `.sha512` URL surfaces as "malformed checksum from %s: %d chars" (`downloader.go:153-159`). Mirror the dedicated error path on the binary side so a future cap bump is signalled by a clear "exceeds N MiB" message rather than a misleading hash mismatch.

Fix: wrap the limit reader so reaching the cap returns a distinct sentinel and the error message names the cap.

S4. **HTTP requests omit a `User-Agent` header** — `pkg/kuberlr/downloader.go:169-184` [Claude Opus 4.7]

```go
req, err := http.NewRequestWithContext(ctx, http.MethodGet, url, http.NoBody)
```

`dl.k8s.io` is operated by SIG Release and accepts the default Go client UA today, but corporate proxies and air-gapped mirrors regularly use UA-based rate limiting or denylists for `Go-http-client/1.1`. A stable UA also helps the SIG Release team correlate kuberlr-style traffic. One header on each request closes the gap.

Fix:

```diff
 req, err := http.NewRequestWithContext(ctx, http.MethodGet, url, http.NoBody)
 if err != nil {
     return err
 }
+req.Header.Set("User-Agent", "rdd-kuberlr/"+componentbasever.Get().GitVersion)
```

S5. **`os.Chmod(tmpName, 0o755)` is a no-op on Windows** — `pkg/kuberlr/downloader.go:116-118` [Claude Opus 4.7]

```go
if err := os.Chmod(tmpName, 0o755); err != nil {
    return err
}
```

Windows file mode is reduced to a single read-only bit; executability is determined by extension (`.exe`) and PE header. `0o755` toggles only the read-only bit (writable, since the write bit is set), so the chmod call has no functional effect. Mode 0755 doesn't break anything, but the next reader of `downloader.go` will read the line as "ensure +x on Windows," which it isn't. A one-line comment or a `runtime.GOOS != "windows"` guard removes the ambiguity.

Fix:

```diff
+// On Windows the executable bit doesn't exist (extension and PE
+// header decide); chmod only affects the read-only bit. Skip the
+// call rather than mislead the next reader.
+if runtime.GOOS != "windows" {
+    if err := os.Chmod(tmpName, 0o755); err != nil {
+        return err
+    }
+}
-if err := os.Chmod(tmpName, 0o755); err != nil {
-    return err
-}
```


## Design Observations

### Concerns

- **Resolver runs against `context.Background()` (future)** [Claude Opus 4.7]. `cmd.Context()` returns `context.Background()` because `cli.RunNoErrOutput` calls `cmd.Execute()`, not `cmd.ExecuteContext()`. The 5 min `httpClient.Timeout` and the 2 s discovery `cfg.Timeout` are the only cancellation backstops; SIGINT to `rdd` during a download kills the process without running the `defer os.Remove(tmpName)`, so a Ctrl-C mid-download leaks a `.kubectl-*` tempfile. Wiring `signal.NotifyContext(context.Background(), os.Interrupt)` in `main` and switching to `cmd.SetContext(ctx)` would close this. Pairs with the existing `TODO(eviction)` at `cache.go:33-39`, which already covers the cleanup side.

- **Cache hit trusts the on-disk binary without re-verification (future)** [Claude Opus 4.7]. `ensureCached` returns immediately when `cachedPath(want)` exists; nothing checks the cached file's sha512 against the upstream digest on hit. Filesystem corruption or a manually-tampered cache file surfaces as a confusing kubectl exec failure rather than a "cache integrity error" the resolver could re-download from. Re-hashing every cached binary on every kubectl invocation is too expensive; a future LRU/TTL sweep (the `TODO(eviction)` at `cache.go:33-39`) is the right place to verify before evicting or reusing.

- **Mirror checksum gives no security against a compromised mirror (future, persistent)** [carry-over from Round 5]. `download` fetches the kubectl binary and its `.sha512` from the same mirror; the hash protects against truncation and disk corruption but not against a mirror serving a malicious binary alongside a matching malicious checksum. SIG Release signs binaries with Sigstore/Cosign. Verifying the Cosign signatures (in addition to or in place of the mirror's own checksum) closes the supply-chain gap. Out of scope for this PR.

- **Hand-rolled subcommand walker duplicates the bound pflag set (future, persistent)** [carry-over from Rounds 3–5]. `parseKubeConfigFlags` already binds the kubectl global flags via `genericclioptions.ConfigFlags.AddFlags`; `isClientOnly` re-walks args manually with the hand-maintained `kubectlGlobalFlagsTakingValue` map. S2 this round is a fresh shape of the same drift class — `--` boundary handling slipping past the manual walker. Driving the walker from the same FlagSet (`fs.Args()` after parse returns the positional tokens) eliminates the map and most of the drift class.

### Strengths

- **Round 5 fixes landed without regression** [Claude Opus 4.7][Codex GPT 5.5]. `environment.md` Path Variables preamble now scopes the no-effect claim correctly; `cmd_service.md` key table includes `cache_dir`; the BATS comment names the right export site; `SkipResolver` uses a package-level bool; `serverVersion` returns `(semver.Version, bool)` with the dead error path gone; the warn-level "downloading kubectl" line surfaces at the default log level.

- **Process replacement on Unix vs. spawn on Windows is the right split** [Claude Opus 4.7]. `syscall.Exec` keeps the kubectl child at the expected place in the process tree, and the Windows path's `*cliexit.Error` envelope plus the empty-message branch in `cmd/rdd/main.go:163-165` propagates the kubectl exit code without a confusing `logrus.Error` line.

- **Same-process toggle plus cross-process env-var recursion guard** [Claude Opus 4.7]. `SkipResolver()` for `rdd ctl` (in-process bool) and `RDD_KUBECTL_RESOLVED` for the rare downloaded-shim re-exec (env-based) cover their respective shapes minimally; the env var no longer leaks into unrelated subprocesses.

- **Hash before mutation; body caps at the right place; reusing kubectl's flag binding** [Claude Opus 4.7][Codex GPT 5.5][Gemini 3.1 Pro]. Round 5 strengths persist: `download` verifies the digest before any chmod or rename; `streamGet` caps the digest body at 4 KiB and the binary at 256 MiB; `parseKubeConfigFlags` binds `genericclioptions.NewConfigFlags(true)` so the resolver and kubectl child target the same cluster by construction.

- **Graceful fall-through on probe failure** [Gemini 3.1 Pro]. `serverVersion` returns `ok=false` for missing kubeconfig, malformed config, unreachable cluster, and unparseable version. Each path falls through to the embedded kubectl rather than aborting the kubectl invocation. Client-only commands keep working when the cluster is unreachable.

- **Robust BATS server fixture** [Gemini 3.1 Pro]. `fake-kube-server` opens its log file with `O_APPEND` so the BATS-side `: >"${LOG_FILE}"` truncation between tests does not race with the server's writes.

**I1. Concurrent download race produces a hard failure on Windows** — `pkg/kuberlr/downloader.go:59-74, 98-119`


```go

		return err
	}

```

`ensureCached` has a TOCTOU window between the `os.Stat` (line 61) and the final `os.Rename` (line 119). When two `rdd kubectl` invocations both miss the cache against the same out-of-skew cluster (a routine pattern: a new-to-the-user remote cluster, two terminals, two scripts), both processes download to their own `.kubectl-*` tempfile and both call `os.Rename(tmp, dst)`.

On Unix this is benign — both renames succeed, the second wins, and the running kubectl child of the loser still resolves through its inode. On Windows (per the repo overview's explicit reminder about `os.Rename`), if process A has already renamed and is mid-`CreateProcess` on the cached binary, process B's rename returns `ERROR_ACCESS_DENIED` / `ERROR_SHARING_VIOLATION` because the file is currently held as an executable image. B then surfaces "resolving kubectl version: …" and exits non-zero even though the file it just wrote is byte-identical to what A produced and what's already on disk.

The race window is wide (the entire download duration — seconds to minutes), so this is not a theoretical edge case for users on slow links.

Fix: when the rename fails, re-check the destination — if it now exists with a matching sha512, treat the rename as "lost the race" and use the existing file. (No new failure modes: if the sha mismatches, fall through to the original error.)

```diff

 		return err
 	}
-	return os.Rename(tmpName, dst)
+	if err := os.Rename(tmpName, dst); err != nil {
+		// Another rdd invocation may have won the race and placed dst
+		// while we were downloading; on Windows os.Rename fails when
+		// the winner's child process is currently holding dst as an
+		// executable image. Accept dst if it is already in place.
+		if _, statErr := os.Stat(dst); statErr == nil {
+			return nil
+		}
+		return err
+	}
+	return nil
```

### Executive Summary

This PR introduces `kuberlr`-style version resolution to `rdd kubectl`, allowing it to dynamically download and execute a version-matched binary when the user's cluster is out of skew with the embedded control plane. The design is robust, handling fallbacks gracefully without blocking offline workflows, but contains a critical panic path when `HOME` is absent and an important race condition on Windows cache population.

### Findings

#### Critical Issues

**C1. `CacheRoot` panics when `os.UserCacheDir` fails** — `pkg/kuberlr/cache.go:51-54` (critical, regression)

```go
	cache, err := os.UserCacheDir()
	if err != nil {
		panic(fmt.Errorf("could not determine user cache directory: %w", err))
	}
```

When `os.UserCacheDir()` fails (e.g., if `$HOME` is unset in a minimal container, systemd service, or headless CI environment), `CacheRoot` panics. Because `CacheRoot` is called by background systems and CLI commands like `rdd svc paths`, this panic will bring down the entire `rdd` daemon or command. A newly-absorbed package must not introduce unrecoverable crash paths.

**Fix:** Fall back to `os.TempDir()` rather than panicking.

```diff
 	cache, err := os.UserCacheDir()
 	if err != nil {
-		panic(fmt.Errorf("could not determine user cache directory: %w", err))
+		cache = os.TempDir()
 	}
 	return filepath.Join(cache, "rancher-desktop")
```

#### Important Issues

**I1. Concurrent downloads fail on Windows due to `os.Rename` race** — `pkg/kuberlr/downloader.go:118-120` (important, regression)

```go
	if err := os.Chmod(tmpName, 0o755); err != nil {
		return err
	}
	return os.Rename(tmpName, dst)
```

On Windows, `os.Rename` does not atomically replace existing files; it fails with `ERROR_ALREADY_EXISTS` / `os.ErrExist` if the destination is present. If multiple `rdd kubectl` invocations race to download the same version concurrently (e.g., via parallel scripts), the first one will succeed, but all others will fail at the rename step and bubble up a download error. Since a valid binary is now present in the cache, the failing processes should just use it.

**Fix:** Catch the error and treat the operation as successful if the destination file now exists.

```diff
 	if err := os.Chmod(tmpName, 0o755); err != nil {
 		return err
 	}
-	return os.Rename(tmpName, dst)
+	if err := os.Rename(tmpName, dst); err != nil {
+		if _, statErr := os.Stat(dst); statErr == nil {
+			return nil
+		}
+		return err
+	}
+	return nil
```

**I2. `isClientOnly` bypasses resolver when `--` is used** — `pkg/kuberlr/resolver.go:266-268` (important, regression)

```go
	for i := 0; i < len(args); i++ {
		arg := args[i]
		if arg == "--" {
			break
		}
```

The `--` separator stops flag parsing in upstream `kubectl`, so an invocation like `kubectl -- get pods` is perfectly valid. However, `isClientOnly` breaks early on `--` and leaves `subcommand` empty. The fallback `if subcommand == "" { return true }` at the end incorrectly classifies the command as client-only. This causes `rdd kubectl -- get pods` to silently skip the version probe and run the embedded binary against the remote cluster. 

**Fix:** If `--` is encountered, grab the next positional argument as the subcommand before breaking.

```diff
 	for i := 0; i < len(args); i++ {
 		arg := args[i]
 		if arg == "--" {
+			if subcommand == "" && i+1 < len(args) {
+				subcommand = args[i+1]
+			}
 			break
 		}
```

### Design Observations

#### Strengths

- **Graceful Error Degradation:** The decision to fail open (falling back to the embedded kubectl) on unreachable clusters, misconfigured kubeconfig files, and garbage HTTP responses is excellent. It preserves local `rdd ctl` capabilities and offline functionality.
- **Developer UX:** Catching the unparseable `v0.0.0-master+$Format:%H$` dev version from `componentbasever` and bypassing the resolver avoids breaking local `make run` cycles for project contributors.
- **Robust BATS Server:** The `fake-kube-server` correctly anticipates test log truncation by opening the log file with `O_APPEND`. This prevents the sparse file bugs commonly seen when external test processes hold file descriptors across `>` shell truncations.

### Testing Assessment

The test suite covers standard cache hits, misses, and mirror failures. The following scenarios are unverified:

1. **`isClientOnly` handling of `--`** (Addresses I2): The test suite checks `exec pod -- completion` (where the subcommand precedes `--`), but misses `--` before the subcommand (e.g., `-- get pods`).
2. **Exit Code Propagation**: `7-kubectl-cache.bats` validates that a checksum mismatch fails (`assert_failure`), but does not assert that `rdd kubectl` correctly propagates the exact non-zero exit code of a downloaded or embedded `kubectl` child process when the probe succeeds. 
3. **Concurrent Download Race**: There are no parallel invocations validating behavior when two processes populate the cache simultaneously (Addresses I1).

### Documentation Assessment

The CLI documentation in `cmd_service.md` and `environment.md` correctly maps out the new paths and cache variables. No further updates needed.

### Acknowledged Limitations

- **Temp file leaks on SIGKILL:** The author acknowledged in a TODO (`pkg/kuberlr/cache.go:33-36`) that process crashes leave `.kubectl-*` temp files stranded in the cache directory, deferring an eviction/sweep pass to the future.
- **No offline cache fallback:** The author acknowledged in a TODO (`pkg/kuberlr/resolver.go:66-69`) that when a mirror is unreachable but a close-enough downloaded binary already exists in the cache, the resolver currently hard-fails rather than using the cached binary.
- **BATS Teardown:** The author added an explicit inline comment (`bats/tests/10-cli/7-kubectl-cache.bats:103-108`) documenting their deliberate deviation from the "no teardown_file" repo rule to avoid leaking ephemeral HTTP server ports across test sessions.
