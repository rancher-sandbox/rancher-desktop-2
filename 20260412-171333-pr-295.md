## Executive Summary

This PR introduces the engine controller that mirrors Docker container engine state (containers, images, volumes) into the `containers.rancherdesktop.io` API group via fullSync-then-watch, forwards user-initiated mirror deletions back to Docker through a finalizer, and dispatches Container start/stop on `spec.state` transitions. It also adds the `rdd set` wait-for-desired-state path, 389 lines of BATS integration coverage, a new CI support-bundle capture wrapper, and Lima template updates (new socket path, rosetta moved under `vmOpts.vz`, distro workaround).

The design is sound: idempotent fullSync, panic-recovery, per-item error aggregation with `errors.Join`, a narrow webhook allowing only `spec.state` transitions, and deliberate reasoning about server-side apply field ownership (documented inline in `sync_containers.go`). Verdict is **merge with fixes**: no critical regressions, four important findings with one-to-three-line fixes, and a cluster of smaller suggestions around condition semantics, clock assumptions, and finalizer edge cases.

Structure: 4 important, 12 suggestions, 0 critical. The important findings are self-healing gaps (panic recovery, watcher-restart early return, sequential finalizer short-circuit, ContainerNamespace resurrection) that the "list-then-watch" architecture would normally handle but for specific early-returns or missing watches.

## Important Issues

I1. **Panic recovery in watcher goroutine doesn't wake the reconciler** â€” `pkg/controllers/app/engine/controllers/docker_watcher.go:117-121` [Claude Opus 4.6, Codex GPT 5.4]

```go
defer func() {
    if r := recover(); r != nil {
        log.Error(nil, "panic in Docker watcher goroutine", "recovered", r)
    }
}()
```

The recover block logs the panic and then `defer close(w.done)` fires, marking the watcher dead. Unlike the normal error-stream exit path at `docker_watcher.go:138-144` â€” which calls `w.enqueueReconcile()` before returning â€” the panic path emits no wake-up signal. The reconciler only learns the watcher died when another reconcile is triggered by an App spec change, a mirror watch event, or the channel source. While the watcher is dead, though, no Container/Image/Volume change will ever fire (the watcher is the source of those events), and an idle user triggers nothing. A panic can silently leave the engine controller in a degraded state for an arbitrarily long time without `ContainerEngineReady` flipping to false.

The comment at `docker_watcher.go:113-116` explicitly claims "The engine reconciler notices the dead watcher via alive() on the next reconcile and starts a fresh one" â€” but there is no guaranteed next reconcile.

Fix: enqueue a reconcile from the panic-recovery defer.

```diff
 defer func() {
     if r := recover(); r != nil {
         log.Error(nil, "panic in Docker watcher goroutine", "recovered", r)
+        w.enqueueReconcile()
     }
 }()
```

Claude additionally suggested capturing `debug.Stack()` so the panic is actionable â€” worth doing.

I4. **Early return after `startWatcher` skips pending specs/finalizers** â€” `pkg/controllers/app/engine/controllers/engine_reconciler.go:142-153` [Gemini 3.1 Pro] (downgraded from critical)

```go

    log.Info("App is running, starting Docker watcher")
    if err := r.startWatcher(ctx); err != nil {
        ...
    }

        return ctrl.Result{}, err
    }
    return ctrl.Result{}, nil

```

After successfully starting a fresh watcher and setting the condition, the reconciler returns without running `reconcileContainerSpecs` or `processFinalizers`. In most cases this is self-healing: `setEngineCondition` flips `ContainerEngineReady` from `Stopped`/`False` to `Connected`/`True`, which triggers a new App reconcile that enters the hot loop. But in the watcher-restart path (the previous condition was already `Connected`/`True` from before the crash), `meta.SetStatusCondition` returns `changed=false`, no status write happens, and no App reconcile fires from that source. `fullSync` then becomes the only trigger: its status Apply calls usually change a timestamp or field and emit watch events, but if Docker state is unchanged during the restart window, the applies can be no-ops and no reconcile follows. Pending user intent (e.g., a `spec.state=running` patch made during the gap) stalls until some unrelated event triggers a reconcile.

Gemini framed this as critical â€” it is real, but the practical blast radius is narrower than a guaranteed stall. Downgrading to important. The fix is trivial and eliminates the worry entirely.

Fix: remove the early return so the reconciler falls through to the shared code path.

```diff

         return ctrl.Result{}, err
     }
-    return ctrl.Result{}, nil
```

## Suggestions

S1. **`since` Docker events filter uses host wall clock, not Docker engine clock** â€” `pkg/controllers/app/engine/controllers/docker_watcher.go:70-76` [Claude Opus 4.6]

```go
// Capture a Docker-relative timestamp before fullSync begins. The
// run() goroutine passes this to the event stream as the Since
// filter, so any mutation that happens while fullSync is snapshotting
// state is replayed once the stream opens.
since := strconv.FormatInt(time.Now().Unix(), 10)
```

The comment claims "Docker-relative timestamp" but `time.Now()` is the **host** clock. The Docker server (inside the Lima VM) interprets the `since` query parameter against its **own** clock. If the host clock is ahead of the guest clock by N seconds, events generated in the window `[now - N, start of event stream]` are filtered out by the server and never reach the watcher â€” silently lost. With Lima/Vz the skew is normally millisecond-scale so practical risk is low, but the failure mode is silent rather than detectable.

Two cheap fixes:

```go
// Option A: bias the timestamp into the past so any realistic skew is covered.
// fullSync is idempotent, so duplicate replay is safe.
since := strconv.FormatInt(time.Now().Add(-2*time.Minute).Unix(), 10)
```

```go
// Option B: query the Docker server's view of "now" before fullSync.
if info, err := cli.Info(ctx); err == nil && info.SystemTime != "" {
    if t, perr := time.Parse(time.RFC3339Nano, info.SystemTime); perr == nil {
        since = strconv.FormatInt(t.Unix(), 10)
    }
}
```

At minimum, fix the comment to say "host wall clock" and acknowledge the clock-skew assumption.

S2. **Container mirror finalizer is not re-added after user removes it** â€” `pkg/controllers/app/engine/controllers/sync_containers.go:117-128` [Claude Opus 4.6]

```go
if apierrors.IsNotFound(err) {
    applyConfig := containersv1alpha1apply.Container(inspect.ID, apiNamespace).
        WithFinalizers(mirrorFinalizer).
        WithSpec(containersv1alpha1apply.ContainerSpec().WithState(containersv1alpha1.ContainerStatusUnknown))
    if err := w.k8s.Apply(ctx, applyConfig, client.FieldOwner(controllerName)); err != nil {
        return fmt.Errorf("failed to create container %s: %w", inspect.ID, err)
    }
} else if err != nil {
    return fmt.Errorf("failed to get container %s: %w", inspect.ID, err)
}
```

The mirror finalizer is set only on the `IsNotFound` (creation) branch. On any subsequent sync of the same Container, the code falls through to the status-only Apply at `sync_containers.go:176` without re-asserting the finalizer. If the user removes the finalizer via `kubectl edit`, the next reconcile won't restore it, and a subsequent user delete bypasses the engine's Docker-side cleanup entirely â€” the K8s mirror is GC'd instantly while the Docker container survives.

`sync_images.go:112-133` and `sync_volumes.go:128-130` reassert the finalizer on every sync because their applies are unconditional. The Container path is the only outlier because of its special "spec-only-on-create" gate.

Fix: split finalizer reassertion into a separate Apply that always runs:

```go

    WithFinalizers(mirrorFinalizer)
if err := w.k8s.Apply(ctx, finalizerOnly,
    client.ForceOwnership, client.FieldOwner(controllerName)); err != nil {
    return fmt.Errorf("failed to assert container finalizer %s: %w", inspect.ID, err)
}
```

With SSA, applying only `WithFinalizers` preserves the user-owned `spec.state` field.

S3. **Concurrent `App.Status().Update` from `AppReconciler` and `EngineReconciler`** â€” `pkg/controllers/app/engine/controllers/engine_reconciler.go:202-214` [Claude Opus 4.6]

```go
func (r *EngineReconciler) setEngineCondition(ctx context.Context, app *appv1alpha1.App, ...) error {
    changed := meta.SetStatusCondition(&app.Status.Conditions, metav1.Condition{...})
    if !changed {
        return nil
    }
    return r.Status().Update(ctx, app)
}
```

The `AppReconciler` at `pkg/controllers/app/app/controllers/app_controller.go:247` already calls `r.Status().Update(ctx, &app)` to mirror LimaVM conditions onto `App.Status.Conditions`. The `EngineReconciler` now also calls `Status().Update` on the same object to set `ContainerEngineReady`. Both reconcilers run in the same manager and are **not** serialized across controllers (controller-runtime serializes within a controller, not across them). When both have a stale local copy and both issue a `Status().Update`, the loser gets a 409 conflict.

In practice conflicts will be infrequent because `meta.SetStatusCondition` short-circuits no-op updates and the App reconciler reconciles less often. But the race is real.

Fix (either of two patterns already used elsewhere in this file):

- Use server-side apply with `client.Apply` + a distinct `FieldOwner` for the engine condition â€” field-level merge avoids resource-version conflict entirely.
- Wrap the `Status().Update` in `retry.RetryOnConflict` plus a re-Get, mirroring `removeMirrorResource` at `docker_watcher.go:271-283` and `deleteAllOfType` at `engine_reconciler.go:262-274`.

```diff
-    return filepath.Join(dir, "docker.sock")
+    return filepath.Join(home, ".rd2", "docker.sock")
```

If Lima does not pre-create it, propagate the error instead of ignoring it.

## Design Observations

### Concerns

- **Reconciler scales linearly per child event (in-scope)** [Claude Opus 4.6]. The in-code comment at `engine_reconciler.go:155-161` already calls this out: every Container/Image/Volume watch event triggers a reconcile via `SetupWithManager`, and each reconcile issues full List() calls across all mirror types. Cost is O(N) per child event. The right long-term fix â€” per-resource reconcilers with watch predicates on `deletionTimestamp set` / `spec.state changed` â€” is the correct structural answer. Worth doing before the controller is expected to handle more than a few dozen containers.
- **Two writers on `App.Status.Conditions` (in-scope)** [Claude Opus 4.6]. The App reconciler mirrors LimaVM conditions; the engine reconciler sets `ContainerEngineReady`. They are different field-level concerns but `Update` sends the entire status. See S3 for the immediate impact. A cleaner long-term design has all App.Status writers use SSA with distinct field owners; this eliminates conflicts entirely and makes per-condition ownership explicit.
- **Engine assumes host/guest clock parity in multiple places (in-scope)** [Claude Opus 4.6]. Beyond the `since` filter (S1), the various `time.Parse(time.RFC3339Nano, ...)` calls in `applyContainer`, `imageStatusFromInspect`, and `applyVolume` all trust Docker's timestamps as-is. None is wrong, but they all make the same assumption. A one-shot startup check (compare `Info().SystemTime` to `time.Now()` and warn if skew exceeds N seconds) would surface clock drift early.

## Executive Summary

This PR introduces an engine controller that mirrors Docker engine state (containers, images, volumes) into the `containers.rancherdesktop.io` API group. The design is sound â€” fullSync-then-watch with finalizer-based deletion forwarding, defensive panic recovery, idempotent state reconciliation, and good test coverage via BATS. **I1. Panic recovery in watcher goroutine doesn't wake the reconciler â€” `pkg/controllers/app/engine/controllers/docker_watcher.go:117-121`**

```go

    if r := recover(); r != nil {
        log.Error(nil, "panic in Docker watcher goroutine", "recovered", r)
    }
}()
```

The deferred recover at lines 117-121 catches the panic and logs it, then the deferred `close(w.done)` at line 112 fires, marking the watcher dead. But unlike the normal error path at line 138-144 (which calls `w.enqueueReconcile()` before returning), the panic path does not enqueue any reconcile. The comment at lines 113-116 explicitly claims "The engine reconciler notices the dead watcher via alive() on the next reconcile and starts a fresh one" â€” but there is no guaranteed next reconcile.

Fix:

```diff

     if r := recover(); r != nil {
         log.Error(nil, "panic in Docker watcher goroutine", "recovered", r)
+        w.enqueueReconcile()
     }
 }()
```

Also consider including a stack trace (`debug.Stack()`) so the panic is actionable. (important, regression)

**S2. `since` Docker events filter uses host wall clock, not Docker engine clock â€” `pkg/controllers/app/engine/controllers/docker_watcher.go:70-76`**

```go
// Capture a Docker-relative timestamp before fullSync begins. The
// run() goroutine passes this to the event stream as the Since
// filter, so any mutation that happens while fullSync is snapshotting
// state is replayed once the stream opens. Without this, the window
// between the List-based snapshot and the event subscription is a
// blind spot during VM startup.
since := strconv.FormatInt(time.Now().Unix(), 10)
```

The comment claims a "Docker-relative timestamp" but `time.Now()` is the **host** clock. The Docker server (running in the Lima VM) interprets the `since` query parameter against its **own** clock (verified in `moby/moby/api/types/time/timestamp.go`'s `GetTimestamp` and the way moby's daemon filters events). With Lima/Vz the skew is normally millisecond-scale, so the practical risk is small, but the comment is misleading and the failure mode is silent rather than detectable.

Two cheap fixes:

```go

// fullSync is idempotent, so duplicate replay is safe.
since := strconv.FormatInt(time.Now().Add(-2*time.Minute).Unix(), 10)
```

or

```go
// Option B: query the Docker server's view of "now" before fullSync.
info, err := cli.Info(ctx)
if err == nil && info.SystemTime != "" {
    if t, perr := time.Parse(time.RFC3339Nano, info.SystemTime); perr == nil {
        since = strconv.FormatInt(t.Unix(), 10)
    }
}
```

At minimum, fix the comment to say "host wall clock" and acknowledge the clock-skew assumption. (suggestion, regression)

## Design Observations

### Concerns

**Reconciler scales linearly per child event (in-scope).** The comment at `engine_reconciler.go:155-161` already calls this out: `reconcileContainerSpecs` and `processFinalizers` each issue full `List()` calls across all mirrored objects on every reconcile, and every Container/Image/Volume watch event triggers a reconcile via `SetupWithManager` at lines 434-436. Cost is O(N) per child event. With hundreds of containers and frequent Docker activity, the engine reconciler will dominate API server time. The right long-term fix described in the comment â€” per-resource reconcilers with watch predicates (`deletionTimestamp set`, `spec.state changed`) â€” is the right structural answer. This refactor is worth doing before this controller hits a workload of more than a few dozen containers.

**Two writers on `App.Status.Conditions` (in-scope).** The `App` reconciler mirrors `LimaVM` conditions (Running, Reconciling, â€¦) and the `Engine` reconciler sets `ContainerEngineReady`, both via `r.Status().Update`. They are different field-level concerns, but Update sends the entire status. This is the same root cause as S6. A cleaner long-term design is to have *all* App.Status writers use server-side apply with distinct field owners; that eliminates conflicts entirely and makes the per-condition ownership explicit.

**The `since` Docker filter is one of three places where the engine assumes host/guest clock parity (in-scope).** The other two are the various `time.Parse(time.RFC3339Nano, ...)` calls in `applyContainer` / `imageStatusFromInspect` / `applyVolume` â€” they parse Docker timestamps as-is, with no clock-correction. None of these are wrong, but they all assume the same thing: the Docker engine clock and the host clock agree. The PR doesn't validate this assumption. Adding a one-shot startup check (compare `Info().SystemTime` to `time.Now()` and warn if skew exceeds N seconds) would surface clock issues early.

### Strengths

- **Idempotent fullSync + watch model**: the fundamental design â€” list-then-watch with full reconciliation on each watcher startup â€” handles every edge case (panic, transient disconnect, intermittent Docker errors) by self-healing. This is the right architecture.
- **Error aggregation pattern**: the per-resource `errors.Join` pattern in `cleanupMirrorResources`, `reconcileContainerSpecs`, and the per-finalizer helpers is a solid choice that ensures one stuck object does not stall the rest. (Modulo I2 above.)
- **Spec-state-once-on-create**: the comment at `sync_containers.go:111-116` explaining why `ForceOwnership` is omitted on the create-Apply is excellent â€” it captures the exact race the author considered and why the chosen design avoids it.
- **Webhook restraint**: the new validator at `container_webhooks.go:35-62` allows only `spec.state` to change, normalizing it to compare other fields. As `ContainerSpec` grows, the comparison will continue to work correctly.
- **`removeMirrorResource` retry-on-conflict**: the watcher-side finalizer strip uses `retry.RetryOnConflict` (`docker_watcher.go:271`) so concurrent reconciles don't require a manual requeue. This matches the pattern used in `deleteAllOfType` and is the right choice.
- **BATS coverage**: the engine.bats file is unusually thorough â€” it covers the dangling/tagged transitions both directions, paused-container state transitions both directions, in-use image deletion, and the "rdd set running=false returns promptly" regression. The comments are likewise thorough about the test design.

### Executive Summary

This PR introduces the engine controller to mirror Docker container engine state (containers, images, and volumes) into Kubernetes resources within the embedded `rdd` control plane, enabling bidirectional sync for lifecycle operations. The review identified two critical regressions: a bash syntax error in the new CI timeout wrapper that will prevent all BATS tests from running, and a stalled reconcile loop when the Docker watcher restarts, which can cause user commands (like deletions or state changes) to be permanently ignored. 

### Findings

#### Critical Issues

C1. **Syntax error in bash script causes timeout wrapper to fail** â€” `scripts/bats-with-timeout.sh:159`, `scripts/bats-with-timeout.sh:349` (critical, regression)

```bash
run_with_timeout() {
    local seconds=$1
    shift
    "$ @" &
```

The bash variable expansion `"$ @"` contains an errant space between the `$` and `@`. In bash, variable expansions cannot contain spaces; this causes the script to evaluate `"$ @"` literally, resulting in a fatal `command not found: $ @` error instead of executing the wrapped target command. Consequently, all BATS test targets running through this wrapper will fail immediately.

Fix: Remove the space to correctly expand the positional parameters:
```diff
-    "$ @" &
+    "$@" &
```

C2. **Engine reconciler stalls indefinitely on watcher restart** â€” `pkg/controllers/app/engine/controllers/engine_reconciler.go:148` (critical, regression)

```go
	if !watcherRunning {
		log.Info("App is running, starting Docker watcher")
		if err := r.startWatcher(ctx); err != nil {
			log.Error(err, "Failed to start Docker watcher")
			_ = r.setEngineCondition(ctx, &app, metav1.ConditionFalse, "ConnectFailed", err.Error())
			return ctrl.Result{}, err
		}
		if err := r.setEngineCondition(ctx, &app, metav1.ConditionTrue, "Connected", "Container engine synced"); err != nil {
			return ctrl.Result{}, err
		}
		return ctrl.Result{}, nil
	}
```

When the Docker watcher is restarted (e.g., after the VM starts or the watcher goroutine crashes), the reconciler initializes the new watcher, performs a full sync, updates the `App` status, and then returns `ctrl.Result{}, nil`. If the user submitted a deletion request or a `spec.state` change while the watcher was offline, this event triggered the current reconcile, but the early return skips the `reconcileContainerSpecs` and `processFinalizers` phases. If the `App` condition was already `Connected`, `setEngineCondition` performs no update, meaning no subsequent watch event is emitted. The reconciler will then stall indefinitely, permanently ignoring the user's operation until an unrelated event triggers another reconcile.

Fix: Remove the early return so the reconciler falls through and processes pending specs and finalizers using the newly started watcher:
```diff
 		if err := r.setEngineCondition(ctx, &app, metav1.ConditionTrue, "Connected", "Container engine synced"); err != nil {
 			return ctrl.Result{}, err
 		}
-		return ctrl.Result{}, nil
 	}
 
 	// reconcileContainerSpecs + processFinalizers each issue one or more
```

#### Important Issues

I1. **Sequential finalizer processing blocks subsequent resource cleanup** â€” `pkg/controllers/app/engine/controllers/engine_reconciler.go:281-289` (important, regression)

```go
	if err := r.processContainerFinalizers(ctx, w); err != nil {
		return err
	}
	if err := r.processImageFinalizers(ctx, w); err != nil {
		return err
	}
	return r.processVolumeFinalizers(ctx, w)
```

The finalizer processing logic short-circuits on the first error encountered. As demonstrated by the integration test (`deleting an in-use Image keeps the finalizer until the container is removed`), deleting an in-use image correctly returns an error and keeps the finalizer in place. However, because `processFinalizers` returns immediately upon this error, `processVolumeFinalizers` is skipped entirely. This means one stuck container or an in-use image will perpetually starve volume deletions across the entire namespace until the blockage is resolved.

Fix: Collect errors across all finalizer types rather than short-circuiting:
```diff
 	if err := r.processContainerFinalizers(ctx, w); err != nil {
-		return err
+		errs = append(errs, err)
 	}
 	if err := r.processImageFinalizers(ctx, w); err != nil {
-		return err
+		errs = append(errs, err)
 	}
-	return r.processVolumeFinalizers(ctx, w)
+	if err := r.processVolumeFinalizers(ctx, w); err != nil {
+		errs = append(errs, err)
+	}
+	return errors.Join(errs...)
```

I2. **Missing GVK causes empty resource kinds in cleanup error logs** â€” `pkg/controllers/app/engine/controllers/engine_reconciler.go:226` (important, regression)

```go
		kind := obj.GetObjectKind().GroupVersionKind().Kind

		retryErr := retry.RetryOnConflict(retry.DefaultRetry, func() error {
// ...
		if retryErr != nil {
			errs = append(errs, fmt.Errorf("failed to remove finalizer from %s/%s: %w",
				kind, obj.GetName(), retryErr))
			continue
		}
```

The `deleteAllOfType` function attempts to extract the resource kind using `GroupVersionKind()`. However, items extracted from a `List` response via client-go `meta.ExtractList` typically have an empty `TypeMeta` (and thus an empty `GroupVersionKind`) because the Kubernetes API server only populates these fields for the top-level list object, not the individual items. This causes the error log to format as `failed to remove finalizer from /volume-name` instead of `Volume/volume-name`, making debugging confusing.

Fix: Derive the type information directly from the concrete object type:
```diff
 		if retryErr != nil {
-			errs = append(errs, fmt.Errorf("failed to remove finalizer from %s/%s: %w",
-				kind, obj.GetName(), retryErr))
+			errs = append(errs, fmt.Errorf("failed to remove finalizer from %T %s: %w",
+				obj, obj.GetName(), retryErr))
 			continue
 		}
```

#### Suggestions

S1. **Watcher startup is fragile to individual container inspection failures** â€” `pkg/controllers/app/engine/controllers/sync_containers.go` (suggestion, regression)

In `syncAllContainers` (and similar image/volume syncs), inspection errors are aggregated but ultimately returned by `fullSync`, which aborts watcher startup and sets `ContainerEngineReady=False`. If a single object permanently fails inspection due to engine-side corruption or race conditions, the watcher will never successfully start, preventing mirroring for all other healthy resources. Consider logging individual inspection errors and skipping the problematic item rather than failing the entire sync loop.
