### Executive Summary

This PR fixes Windows process signaling by placing hostagents in their own process groups (`CREATE_NEW_PROCESS_GROUP`), adds WSL2 deadlock prevention (`wsl --terminate` before `--unregister`), pre-generates SSH keys to sidestep MSYS2 path conversion, and switches CI from WSL2-based to MSYS2-based BATS. Two important issues remain: `stopInstanceForcibly` can kill recycled PIDs when called on stopped instances, and `killHostagent` only kills the parent process, leaving children alive on Windows.

### Important Issues

1. **Stopped-instance force-stop can kill recycled PIDs on Windows** â€” `pkg/controllers/lima/limavm/controllers/limavm_lifecycle.go:47-56` (important, regression) [Codex GPT 5.4]

```go
// Always force-stop before deleting, even if the instance appears
// stopped.

// Clear stale PIDs so Lima's Delete â†’ StopForcibly does not kill
// unrelated processes if the PIDs were recycled (likely on Windows).
existingInst.DriverPID = 0
existingInst.HostAgentPID = 0
```

The `deleteInstance` path calls `stopInstanceForcibly` at line 52 before zeroing the PID fields at lines 55â€“56. For a stopped instance on Windows, Lima's `ReadPIDFile` returns a non-zero PID if the PID file exists and the OS recycled that PID to a new process (`store/instance.go:210-211` returns early for any running Windows process). `stopInstanceForcibly` then calls `process.KillTree` at line 598 with this recycled PID, killing an unrelated process tree. The zeroing at lines 55â€“56 prevents Lima's subsequent `Delete â†’ StopForcibly` from repeating the damage but does not prevent the first kill.

On Unix this cannot happen: `ReadPIDFile` sends signal 0 to verify ownership and removes stale PID files for dead processes (`store/instance.go:213-217`). On Windows, `os.FindProcess` succeeds for any running PID regardless of ownership.

The intent of the unconditional `stopInstanceForcibly` call is to run `wsl --terminate` before `--unregister`. PID-based killing is unnecessary for stopped instances.

Fix: split WSL2-specific teardown from PID-based killing for stopped instances:

```diff
- stopInstanceForcibly(ctx, logger, existingInst)
+ if existingInst.Status == limatype.StatusRunning || existingInst.Status == limatype.StatusBroken {
+     stopInstanceForcibly(ctx, logger, existingInst)
+ } else if existingInst.VMType == limatype.WSL2 {
+     // A "stopped" WSL2 distro can retain kernel state that deadlocks
+     // wsl.exe --unregister. Terminate it without PID-based killing,
+     // since the PIDs may have been recycled on Windows.
+     wslCtx, cancel := context.WithTimeout(ctx, 10*time.Second)
+     defer cancel()
+     distroName := "lima-" + existingInst.Name
+     _ = exec.CommandContext(wslCtx, "wsl.exe", "--terminate", distroName).Run()
+ }
  existingInst.DriverPID = 0
```

```go
		return
```

`Process.Kill()` at line 222 sends `TerminateProcess` on Windows, which kills only the parent hostagent. Child processes (SSH port forwarders) inherit the parent's stdout/stderr pipes and keep them open. The subsequent `forceStop()` at `limavm_lifecycle.go:529` calls `process.KillTree` with the inst's `HostAgentPID`, but on Windows `taskkill /F /T /PID` returns exit code 128 (process not found) for the already-dead parent and does not traverse to its children. The children survive, `cmd.Wait()` blocks on the held pipes, and the reaper goroutine at `hostagent_watcher.go:83-91` leaks.

Fix: replace `Process.Kill()` with `process.KillTree` to kill the entire process group while the parent is still alive:

2. **`KillTree` semantics differ between Unix and Windows** â€” `pkg/util/process/process_unix.go:39`, `pkg/util/process/process_windows.go:66` (suggestion, gap) [Gemini 3.1 Pro]

#### Important Issues

1. **`killHostagent` leaves child processes orphaned and can leak the watcher goroutine** â€” `pkg/controllers/lima/limavm/controllers/hostagent_watcher.go:217` (important, regression)

```go
func (r *LimaVMReconciler) killHostagent(name string) {
	// ...
	if !ok || state.cmd == nil || state.cmd.Process == nil {
		return
	}
	_ = state.cmd.Process.Kill()
}
```

`killHostagent` calls `state.cmd.Process.Kill()`, which only terminates the parent hostagent process. If the hostagent spawned children (e.g. SSH port forwarders) that inherit its stdout/stderr pipes, `cmd.Wait()` in the watcher goroutine will block indefinitely until those children exit.

While `shutdownHostagent` attempts to mitigate this by immediately calling `forceStop()` (which uses `process.KillTree`), `forceStop()` relies on reading PIDs from disk via `store.Inspect` if `inst` is nil. If `store.Inspect` fails (e.g., missing PID files), `KillTree` is skipped. The child processes are left running, and the watcher goroutine permanently leaks.

Fix: Since `killHostagent` already has the authoritative process group ID in `state.cmd.Process.Pid`, it should use `process.KillTree` directly to guarantee the whole process group is terminated regardless of disk state.

```go
func (r *LimaVMReconciler) killHostagent(name string) {
	// ...
	if !ok || state.cmd == nil || state.cmd.Process == nil {
		return
	}
	ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
	defer cancel()
	_ = process.KillTree(ctx, state.cmd.Process.Pid)
}
```

2. **`process.Interrupt` fails across consoles on Windows, leading to hard-kill** â€” `pkg/service/cmd/service.go:345` (important, gap)

```go
	if err := process.Interrupt(pid); err != nil {
		if err := process.Kill(pid); err != nil {
			return fmt.Errorf("failed to stop %q control plane with pid %d: %w", instance.Name(), pid, err)
		}
	}
```

`StopWithWait` uses `process.Interrupt(pid)` (which on Windows calls `GenerateConsoleCtrlEvent`) to gracefully shut down the service. However, `GenerateConsoleCtrlEvent` only delivers the signal if the calling process (`rdd svc stop`) shares the same Windows console as the target process group (`rdd svc start` or daemon).

If invoked from a different terminal window or if the daemon is detached, `GenerateConsoleCtrlEvent` fails. The code then falls back to `process.Kill` (`TerminateProcess`), instantly hard-killing the daemon. Because the hostagents now run in their own process groups (via `process.SetGroup`), `TerminateProcess` on the parent does not kill them, leaving them orphaned in the background (though they will self-heal upon next restart).

Fix: Acknowledge this limitation in a comment. A true cross-console graceful shutdown on Windows will require a different IPC mechanism (e.g., a local HTTP endpoint or named pipe).

3. **`process.KillTree` semantics differ between Unix and Windows for processes with `SetGroup`** â€” `pkg/util/process/process_unix.go:28` (important, gap)

```go
func KillTree(_ context.Context, pid int) error {
	err := unix.Kill(-pid, unix.SIGKILL)
```

Unix `unix.Kill(-pid, ...)` only kills the process group. If the daemon starts hostagents with `SetGroup` (placing them in a new process group), `KillTree` on the daemon's PID will *not* kill the hostagents on Unix.

Conversely, on Windows, `taskkill /T` walks the Parent Process ID (PPID) tree and *will* kill all children, completely ignoring process groups. When `StopWithWait` times out and calls `KillTree` on the daemon, Unix leaves hostagents running (to be recovered as orphans on next start), while Windows force-kills them abruptly.

Fix: Document this cross-platform semantic difference on the `KillTree` function. Given the robust orphan recovery built into the controller, standardizing the behavior may not be strictly necessary, but the abstraction leak should be documented.
