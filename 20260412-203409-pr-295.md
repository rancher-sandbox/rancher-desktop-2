## Executive Summary

This round adds an engine controller that mirrors Docker state (`Container`, `Image`, `Volume`) into the rdd Kubernetes API, forwards user-initiated deletes back to Docker, and makes `rdd set` wait for the desired state by default. The core structure ŌĆö `dockerWatcher` for Docker-side I/O, `EngineReconciler` for controller-runtime integration ŌĆö is clean and carefully commented, and the BATS coverage is thorough. Review round 3's findings have been addressed.

Structure: 0 critical, 4 important, 13 suggestions, 3 design observations.

The dominant issue is a single line of state-matching logic in `reconcileContainerState`: once a `Container` with `spec.state=created` reaches Docker's `exited` state, `desired != actual` keeps dispatching `ContainerStop` on every reconcile. This was independently caught by all three agents. Three other Important findings surface real concurrency or drift bugs that happen to be structural rather than catastrophic: the App controller's pre-existing `Status().Update` is no longer safe now that the engine controller writes the same conditions, `rdd set`'s watch predicate ignores the `ObservedGeneration` the engine already stamps, and the `status.ports` list is built in non-deterministic map-iteration order which churns atomic-list SSA applies.

## Important Issues

I1. **`reconcileContainerState` dispatches redundant `ContainerStop` every reconcile once a `created`-desired container reaches `exited`** ŌĆö `pkg/controllers/app/engine/controllers/docker_watcher.go:334-370` [Claude Opus 4.6, Codex GPT 5.4, Gemini 3.1 Pro]

```go
actual := c.Status.Status
if desired == actual {
    return nil
}
// ...
switch desired {
case containersv1alpha1.ContainerStatusCreated:
    log.Info("Stopping container", "id", c.Name)
    _, err := w.cli.ContainerStop(ctx, c.Name, dockerclient.ContainerStopOptions{})
    return err
}
```

After the user patches `spec.state=created` on a running container, `ContainerStop` transitions the container to Docker's `exited` state (not `created`), and `applyContainer` writes `status.status=exited` at `sync_containers.go:157`. On the next reconcile ŌĆö triggered by any `Container`/`Image`/`Volume` watch event via `SetupWithManager`'s unconditional enqueueApp mapping ŌĆö `desired == "created"` and `actual == "exited"`, so the `desired != actual` check at line 341 falls through and dispatches `ContainerStop` again. Docker's `ContainerStop` on a non-running container returns `NotModified` which the client treats as success, so no error propagates ŌĆö but the reconciler never reaches a quiescent state for this container. Cost scales with the rate of unrelated watch events times the number of `created`-desired stuck containers. The same issue applies to `actual == "dead"` or `actual == "created"` when `desired == "created"`.

The new BATS test at `engine.bats:231-248` only verifies the one-shot transition to `exited` and does not observe the subsequent churn, so this isn't caught by CI.

Fix: treat any stopped state (`created`, `exited`, `dead`) as satisfying `desired=created`, symmetric with the unpause handling in the `Running` branch at lines 354-363.

```diff
 func (w *dockerWatcher) reconcileContainerState(ctx context.Context, c *containersv1alpha1.Container) error {
     desired := c.Spec.State
     if desired == containersv1alpha1.ContainerStatusUnknown {
         return nil
     }

     actual := c.Status.Status
-    if desired == actual {
-        return nil
-    }
+    // Treat any stopped state as satisfying desired=created.
+    if desired == containersv1alpha1.ContainerStatusCreated {
+        switch actual {
+        case containersv1alpha1.ContainerStatusCreated,
+            containersv1alpha1.ContainerStatusExited,
+            containersv1alpha1.ContainerStatusDead:
+            return nil
+        }
+    }
+    if desired == actual {
+        return nil
+    }
```

(Gemini flagged this as Critical on the grounds of "Docker API spam loop"; Claude and Codex called it Important. Calibrated as Important here: the loop is bounded by unrelated event frequency, `NotModified` is a silent success, and no error propagates ŌĆö the cost is wasted RPCs, not a hang or failure.)

I2. **`App.Status.Conditions` now has two writers but the AppReconciler still uses a blind `Status().Update`** ŌĆö `pkg/controllers/app/app/controllers/app_controller.go:229-251` [Codex GPT 5.4]

```go
// Mirror LimaVM status conditions onto the App status.
// The priority chain above returns after every other action, so the App's
// resourceVersion from the initial Get is still current ŌĆö no re-read needed.
// ...
if statusChanged {
    if err := r.Status().Update(ctx, &app); err != nil {
        log.Error(err, "Unable to update App status")
        return ctrl.Result{}, err
    }
}
```

This PR adds a second writer to `App.Status.Conditions` via `setEngineCondition` (`engine_reconciler.go:223-240`), which correctly wraps its update in `retry.RetryOnConflict` with a re-Get. The existing `AppReconciler` still does not: the comment at line 230-231 claims the `resourceVersion` from the initial Get is still current, which was true when the App controller was the sole writer but is no longer true once the engine reconciler can land a `ContainerEngineReady` update in between the Get and the Update. The repo overview explicitly calls this out: *"`App.Status.Conditions` has multiple concurrent writers (`AppReconciler` mirrors LimaVM conditions; `EngineReconciler` writes `ContainerEngineReady`). Flag any non-SSA, non-retry-on-conflict `Status().Update` on the App object."*

This isn't catastrophic ŌĆö controller-runtime requeues on the returned conflict error and the next reconcile re-Gets ŌĆö but it turns routine status churn into avoidable 409s and noisy reconcile-error logs.

Fix: convert the App controller's status write to the same `retry.RetryOnConflict` + re-Get pattern used in `setEngineCondition`.

```go
if statusChanged {
    if err := retry.RetryOnConflict(retry.DefaultRetry, func() error {
        latest := &v1alpha1.App{}
        if err := r.Get(ctx, client.ObjectKeyFromObject(&app), latest); err != nil {
            return err
        }
        for _, cond := range limaVM.Status.Conditions {
            apimeta.SetStatusCondition(&latest.Status.Conditions, metav1.Condition{
                Type:               cond.Type,
                Status:             cond.Status,
                Reason:             cond.Reason,
                Message:            base.TruncateConditionMessage(cond.Message),
                ObservedGeneration: latest.Generation,
                LastTransitionTime: cond.LastTransitionTime,
            })
        }
        return r.Status().Update(ctx, latest)
    }); err != nil {
        log.Error(err, "Unable to update App status")
        return ctrl.Result{}, err
    }
}
```

I3. **`rdd set running=true` wait ignores the `ObservedGeneration` the engine already stamps** ŌĆö `cmd/rdd/set.go:453-474`, `pkg/controllers/app/engine/controllers/engine_reconciler.go:223-240` [Claude Opus 4.6]

```go
if runningVal == "true" {
    logrus.Info("Waiting for container engine to be ready")
    return watchCondition(ctx, config, func(obj *unstructured.Unstructured) bool {
        return conditionStatus(obj, "ContainerEngineReady") == metav1.ConditionTrue
    })
}
```

The predicate only checks `Status == True` and has no generation handoff from the preceding patch. The "Known hazard" comment at `cmd/rdd/set.go:445-452` acknowledges that `rdd set containerEngine.name=<other> running=true` on an already-running VM can be satisfied by the stale `Connected` condition left over from the previous backend ŌĆö but `setEngineCondition` already writes `ObservedGeneration: latest.Generation` at `engine_reconciler.go:234`, so the fix is cheap: capture the generation from the patch response in `setAction` and compare it in the predicate. The `waitForDesiredState` signature needs a `minGen int64` parameter, `patchApp`/`createAndPatchApp` need to return the post-patch generation, and `watchCondition`'s predicate gains `cond.ObservedGeneration >= minGen`.

Without the fix, `rdd set containerEngine.name=containerd` (without changing `running`) is the worst case: no wait triggers at all because no `running` argument is set, even though a full VM restart follows. With `running=true` included, the wait happens but can be satisfied by a pre-patch snapshot.

Fix sketch (conceptual, paired with I4 in `rdd set.go`):

```go
// In setAction, after patchApp/createAndPatchApp:
var app appv1alpha1.App
if err := c.Get(ctx, client.ObjectKey{Name: "app"}, &app); err != nil {
    return fmt.Errorf("failed to get App after patch: %w", err)
}
minGen := app.Generation

// In waitForDesiredState predicate:
return conditionStatusForGeneration(obj, "ContainerEngineReady", metav1.ConditionTrue, minGen)
```

Claude flagged this as Important because the fix is already available (the stamp exists), not because the hazard itself is new.

I4. **`status.ports` is built in non-deterministic map-iteration order and churns the atomic list on every sync** ŌĆö `pkg/controllers/app/engine/controllers/sync_containers.go:176-190` [Claude Opus 4.6]

```go

    for portName, ports := range inspect.NetworkSettings.Ports {
        var bindings []*containersv1alpha1apply.ContainerPortBindingApplyConfiguration
        for _, port := range ports {
            bindings = append(bindings, containersv1alpha1apply.ContainerPortBinding().
                WithHostIP(port.HostIP.String()).
                WithHostPort(port.HostPort))
        }
        applyPorts = append(applyPorts, containersv1alpha1apply.ContainerPort().
            WithName(portName.String()).
            WithBindings(bindings...))
    }
}
```

`inspect.NetworkSettings.Ports` is a Go map, so iteration order is randomized per process. The Container CRD declares `status.ports` as a plain array with no `x-kubernetes-list-type` annotation (verified against `pkg/controllers/containers/container/crd.yaml`: only `status.conditions` carries `x-kubernetes-list-type: map`). Lists without a list-type default to atomic, which is position-sensitive: the apiserver treats `[A, B]` and `[B, A]` as different values and produces a new `resourceVersion` on every apply that reorders them. That fires a watch event, which the engine reconciler catches via `Watches(&containersv1alpha1.Container{}, enqueueApp)`, which re-enters `reconcileContainerSpecs + processFinalizers` ŌĆö a four-list sweep per watch event. For a container with two or more exposed ports, every unrelated Docker event (image pull, volume create, etc.) can amplify into a status re-apply and a full reconcile sweep.

Fix: sort `inspect.NetworkSettings.Ports` by port name before building the apply slice. The inner `bindings` loop is Docker-ordered (IPv4 then IPv6) and does not need sorting unless Docker's internal order changes.

```diff

+    for p := range inspect.NetworkSettings.Ports {

+    }
+    sort.Slice(portNames, func(i, j int) bool { return portNames[i] < portNames[j] })
-    for portName, ports := range inspect.NetworkSettings.Ports {
+    for _, portName := range portNames {
+        ports := inspect.NetworkSettings.Ports[portName]
```

Alternatively, annotate `Ports` with `+listType=map` + `+listMapKey=name` in `container_types.go` so the apiserver merger ignores ordering ŌĆö this is more robust but requires a CRD regeneration and is a larger change.

## Suggestions

S1. **`DockerSocket()` silently swallows `os.MkdirAll` error** ŌĆö `pkg/instance/instance.go:136-144` [Claude Opus 4.6]

```go
var DockerSocket = sync.OnceValue(func() string {
    home, err := os.UserHomeDir()
    if err != nil {
        panic(fmt.Errorf("could not get home directory: %w", err))
    }
    dir := filepath.Join(home, ".rd2")
    _ = os.MkdirAll(dir, 0o700)
    return filepath.Join(dir, "docker.sock")
})
```

If `MkdirAll` fails (permission denied, read-only filesystem), the error is discarded and the returned path may point at a non-existent parent. `dockerclient.WithHost("unix://" + socketPath)` in `docker_watcher.go:47` then fails at connect time with a cryptic "no such file or directory". Sibling helpers (`LogDir`, `ShortDir`, etc.) don't do `MkdirAll` ŌĆö the directory is really Lima's responsibility, and creating it here is inconsistent.

Fix: drop the `MkdirAll` call entirely and let Lima own the directory lifecycle.

S2. **`setEngineCondition` does not tolerate App deletion during retry** ŌĆö `pkg/controllers/app/engine/controllers/engine_reconciler.go:223-241` [Claude Opus 4.6]

```go
return retry.RetryOnConflict(retry.DefaultRetry, func() error {
    latest := &appv1alpha1.App{}
    if err := r.Get(ctx, client.ObjectKey{Name: app.Name}, latest); err != nil {
        return err
    }
```

`Reconcile` wraps its initial Get in `client.IgnoreNotFound`, but if the App is deleted between that Get and a later `setEngineCondition` call (e.g., `rdd svc delete` runs concurrently), the inner Get returns `NotFound` and `retry.RetryOnConflict` does not treat `NotFound` as retriable ŌĆö the error propagates up and the reconciler returns it, causing a gratuitous requeue-then-bail.

Fix: treat `NotFound` as a no-op inside the retry closure.

```diff
 return retry.RetryOnConflict(retry.DefaultRetry, func() error {
     latest := &appv1alpha1.App{}
     if err := r.Get(ctx, client.ObjectKey{Name: app.Name}, latest); err != nil {
+        if apierrors.IsNotFound(err) {
+            return nil
+        }
         return err
     }
```

S3. **`deleteImage` may untag the wrong image in a tag-race** ŌĆö `pkg/controllers/app/engine/controllers/docker_watcher.go:393-404` [Claude Opus 4.6]

```go
ref := img.Status.ID
if img.Status.RepoTag != "" {
    ref = img.Status.RepoTag
}
_, err := w.cli.ImageRemove(ctx, ref, dockerclient.ImageRemoveOptions{})
```

Consider: (1) Image A is tagged `busybox:latest`, mirror A holds `Status.RepoTag=busybox:latest`. (2) `docker pull busybox:latest` brings in image B, moves the tag, emits untag(A)+tag(B). (3) Before the watcher processes those events, the user deletes mirror A. `processImageFinalizers` reads the stale mirror A, calls `ImageRemove("busybox:latest")`, which now hits image B and untags it. The ID-based path would have untagged A correctly.

The intent of using the tag is presumably to untag-only rather than remove ŌĆö so that a multi-tagged image keeps its other tags. A safer approach re-inspects the image and verifies `Inspect().RepoTags` still contains the tag before calling `ImageRemove`, or removes by `status.ID` + a `ForceOwnership: false` flag.

S4. **Image finalizer path has no empty-status guard, asymmetric with the Volume path** ŌĆö `pkg/controllers/app/engine/controllers/engine_reconciler.go:402-424` [Claude Opus 4.6]

`processVolumeFinalizers` at lines 426-455 guards `w.deleteVolume(ctx, v.Status.Name)` with `if v.Status.Name != ""` because a bare-skeleton mirror has no engine name to forward to. The Image path has no equivalent guard: if a user directly creates an `Image` mirror via kubectl (with the mirror finalizer) and then deletes it before status is populated, `deleteImage` passes an empty string to `ImageRemove`, which the daemon rejects. Since the Container webhook only intercepts Update not Create, such a bare mirror is technically reachable.

Fix: symmetrically guard the Image path with `img.Status.ID != "" || img.Status.RepoTag != ""`.

S5. **`parseContainerName("/")` returns `("moby", "")` which downstream CRD validation would reject** ŌĆö `pkg/controllers/app/engine/controllers/sync_containers.go:207-214` [Claude Opus 4.6]

The table test at `sync_containers_test.go:42-51` pins the `"/"` result as `("moby", "")`. That flows into `ContainerStatus().WithName("")` at `sync_containers.go:151`, and `status.name` is `+required` in `container_types.go:73-75`, so the apiserver would reject the apply with a validation error. In practice `inspect.Name` from Docker is never `/` because Docker always generates a name, so this is defensive only. But the current test pins a result that in the hot path would produce an invalid mirror.

Fix: return an explicit error or fall back to a synthetic name (e.g., the container ID). A test assertion + a clear error is better than silently producing an invalid apply.

S6. **`applyContainer` reasserts the finalizer even on a terminating mirror** ŌĆö `pkg/controllers/app/engine/controllers/sync_containers.go:135-147` [Claude Opus 4.6]

If a user has initiated `kubectl delete container/...` (mirror has `DeletionTimestamp`) and Docker then emits a container event (the container dies on its own), `applyContainer` reaches the `else` branch and tries to re-assert the mirror finalizer via SSA. Adding a finalizer to a deleting object is typically a no-op under SSA (the finalizer is still present), but if any other field manager stripped it the reassertion fails with a cryptic error and the next Docker event for the same container repeats the failure.

Fix: skip the finalizer reassertion when `existing.DeletionTimestamp != nil`.

S7. **Docker state mapping assumes the moby enum is a strict subset of the CRD enum** ŌĆö `pkg/controllers/app/engine/controllers/sync_containers.go:157` [Claude Opus 4.6]

```go
WithStatus(containersv1alpha1.ContainerStatusValue(inspect.State.Status)).
```

`inspect.State.Status` is a free-form Docker string, directly cast into the CRD enum without validation. The CRD enum at `container_types.go:11` currently covers every documented Docker state, but a future Docker release that introduces a new state string (`initializing`, `error`, etc.) would cause SSA applies to fail with CRD validation errors, the per-container error would be logged and skipped by `syncAllContainers:57-59`, and that container's mirror would silently fall behind.

Fix: whitelist the known states and fall back to `unknown` on mismatch, so forward-compat with Docker is explicit rather than implicit.

S8. **`setAction` does not hand off the post-patch generation to `waitForDesiredState`** ŌĆö `cmd/rdd/set.go:293-311` [Claude Opus 4.6]

Pairs with I3: neither `createAndPatchApp` nor `patchApp` returns the `metadata.generation` of the post-write App object, so `waitForDesiredState` has no way to filter stale condition snapshots in the informer's initial list. The fix is a small plumbing change ŌĆö return `app.Generation` from the patch path and thread it into `watchCondition`'s predicate.

S9. **Gemini: Missing channel closure check in Docker watcher loop** ŌĆö `pkg/controllers/app/engine/controllers/docker_watcher.go:183-188` [Gemini 3.1 Pro]

## Design Observations

### Concerns

1. **Engine reconciler iterates all containers on every unrelated watch event** ŌĆö `pkg/controllers/app/engine/controllers/engine_reconciler.go:170-182` [Claude Opus 4.6, Codex GPT 5.4] `(future)`. The author's TODO at lines 170-176 acknowledges this. Concrete cost: every `Container`/`Image`/`Volume` event triggers a full `reconcileContainerSpecs + processFinalizers` sweep (4 List calls + O(N) Docker RPCs in the stuck-state case ŌĆö see I1). Splitting into per-kind reconcilers with watch predicates (`deletion timestamp set`, `spec.state changed`) would bound cost to O(1) per event. Not blocking, but I1 is a direct consequence of this design.

2. **Engine namespace and `containernamespace/moby` are not repaired after user-initiated delete** ŌĆö `pkg/controllers/app/engine/controllers/engine_reconciler.go:457-475`, `pkg/controllers/app/engine/controllers/sync_volumes.go:56-66` [Codex GPT 5.4] `(in-scope)`. `ensureNamespace` and `syncContainerNamespace` run only in `fullSync` at watcher startup; `SetupWithManager` does not watch `corev1.Namespace` or `ContainerNamespace`. A user delete of either resource is therefore not repaired until the Docker watcher is torn down and restarted. For the moby `ContainerNamespace` this is documented at `sync_volumes.go:56-61` as a deliberate design choice (to avoid a finalizer trap), and the BATS test at `engine.bats:355-361` codifies the "stays deleted" behavior. For the `rancher-desktop` Namespace there is no such deliberation ŌĆö if deleted, subsequent mirror applies would fail with NotFound until the next watcher restart. Users are unlikely to delete these resources in an embedded single-user context, so Codex's Important severity is demoted to a design observation; but a `Watches(&corev1.Namespace{}, enqueueAppIfName("rancher-desktop"))` would close the gap cleanly.

3. **`Container.Status` has concurrent SSA and non-SSA writers** ŌĆö `pkg/controllers/containers/container/container_controller.go` (pre-PR) and `pkg/controllers/app/engine/controllers/sync_containers.go` (new) [Claude Opus 4.6] `(future)`. The pre-existing container reconciler writes `status.Conditions` via a plain `r.Status().Update()` without a field manager, while the new engine controller writes `status.Name`, `status.Status`, `status.Ports`, etc. via SSA with field manager `engine-controller`. Each `Status().Update` sends the full status object, so the container reconciler's client implicitly takes ownership of engine-written fields, which the engine controller then forcibly reclaims via `ForceOwnership` on the next sync. Functional today, but the repo-context rule about concurrent writers on `App.Status.Conditions` applies symmetrically here. Migrating the container reconciler to SSA with a distinct field manager would eliminate the ping-pong.

### Strengths

- **Clean separation between `dockerWatcher` and `EngineReconciler`** [Claude Opus 4.6]. Docker-side I/O and controller-runtime integration are split cleanly, making ownership of state obvious.
- **Daemon-clock `Since` filter is the right design choice** [Claude Opus 4.6, Codex GPT 5.4, Gemini 3.1 Pro]. `dockerEventsSince()` uses Docker's `Info().SystemTime` to anchor the event stream's `Since` filter rather than host `time.Now()`, avoiding the host/guest clock-skew bug that would otherwise drop startup-window events. The host-clock fallback with a 2-minute bias is pragmatic for the rare case where `Info` is unavailable.
- **Race/retry decisions are commented with reasoning, not just mechanics** [Claude Opus 4.6]. `sync_containers.go:102-122` on why `create` omits `ForceOwnership` is a textbook example ŌĆö it captures the "fail loudly on concurrent user patch" intent that a terse SSA snippet would hide.
- **`errors.Join` throughout cleanup/sync paths** [Claude Opus 4.6]. `cleanupMirrorResources`, `processFinalizers`, `syncAllContainers`, and `syncAllImages` consistently avoid the "one stuck object blocks the rest" footgun.
- **`NotApplicable=True` for containerd backend** [Gemini 3.1 Pro]. Reporting `ContainerEngineReady=True` with Reason `NotApplicable` is a clean way to let `rdd set running=true containerEngine.name=containerd` complete its wait even though mirroring is a no-op ŌĆö with the caveat about Reason discrimination captured in I3.

## Design Observations

### Concerns

**Cross-controller status writes on Container are not coordinated.** The existing `container_reconciler.go` (pre-PR) writes `status.Conditions` via `r.Status().Update()` without a field manager, and the new engine controller writes `status.Name`, `status.Status`, `status.Ports`, etc. via SSA with field manager `engine-controller`. Both controllers reconcile the same object on every watch event. A regular `Status().Update()` sends the full status object, which means the container reconciler's client can implicitly take ownership of the engine-written fields (field manager is the client name, not `engine-controller`). The subsequent engine-controller apply then has to forcibly reclaim those fields via `ForceOwnership` at `sync_containers.go:195-196`. This works, but each Container update bounces between the two field managers and can generate 2├Ś the reconciles. Per the repo overview: "`App.Status.Conditions` has multiple concurrent writers... Flag any non-SSA, non-retry-on-conflict `Status().Update` on the App object" ŌĆö the same logic applies to `Container.Status.Conditions`. Migrating `container_reconciler.go` to SSA with a distinct field manager (`container-condition-reconciler` or similar) would eliminate the churn. `(future)`

**Engine reconciler iterates all containers on every unrelated watch event.** `engine_reconciler.go:170-182` acknowledges this with a TODO. The concrete amplification depends on how many containers have a non-`unknown` spec.state. For the current test fixtures (a handful of containers), it's fine; in a developer's real environment with 30-50 containers, every Container/Image/Volume event fires a full `reconcileContainerSpecs + processFinalizers` sweep, which is `4 ├Ś O(N)` Lists followed by `O(N)` Docker calls in the stuck-state case (see I1). A per-type reconciler with predicates would bound cost to `O(1)` per event. `(future)`

**Event `Since` filter uses second-precision, host-clock fallback trades correctness for robustness.** `docker_watcher.go:78-90` fetches Docker's `SystemTime` and falls back to `time.Now().Add(-2*time.Minute)` on failure. The 2-minute bias absorbs arbitrary clock skew, which is pragmatic. One edge case: if the host clock is *ahead* of the guest clock by more than 2 minutes, the fallback window misses real events. Since Lima's clock is synced via chrony on boot, this is unlikely ŌĆö but is worth a sanity check on startup. `(future)`

### Strengths

- The split between `docker_watcher` (Docker client + event handling) and `engine_reconciler` (controller-runtime integration + mirror lifecycle) is clean and makes the ownership of state obvious. Good separation of concerns.
- Every non-trivial race/retry decision is commented in place with the reasoning, not just the mechanic. `sync_containers.go:102-122`'s explanation of why create uses no `ForceOwnership` is a textbook example.
- Error handling via `errors.Join` in `cleanupMirrorResources`, `processFinalizers`, and `syncAllContainers` consistently avoids the "one stuck object blocks the rest" footgun.
- The `imageStatusFromInspect` helper explicitly documents the slice-aliasing trap at `sync_images.go:146-150` ŌĆö this is the kind of subtle Go pitfall that bites later, and catching it upfront is valuable.
- The BATS test suite covers the non-obvious cases (untag-leaves-dangling, tagging-dangling-removes-mirror, in-use image finalizer, spec.state paused-to-created, uppercase volume names) that a less thoughtful test would miss.

1. **Repeated reconcile after `spec.state=created` with Docker in `exited` state (I1)** ŌĆö no test verifies that the reconciler doesn't call ContainerStop on every subsequent event. ### Executive Summary

This PR implements the engine controller that connects to the Docker socket, mirrors container engine state (Containers, Images, Volumes) into Kubernetes resources, and supports basic bidirectional state management. The core sync and event loop are well-structured, but there is a critical logical flaw where the controller fails to recognize that a container is already stopped, leading to infinite API spam loops. 

### Findings

#### Critical Issues

**C1. Docker API spam loop on stopped containers** ŌĆö `pkg/controllers/app/engine/controllers/docker_watcher.go:274-301` (critical, regression)

```go
	actual := c.Status.Status
	if desired == actual {
		return nil
	}

	log := logf.FromContext(ctx).WithName("docker-watcher")

	switch desired {
	// ...
	case containersv1alpha1.ContainerStatusCreated:
		log.Info("Stopping container", "id", c.Name)
		_, err := w.cli.ContainerStop(ctx, c.Name, dockerclient.ContainerStopOptions{})
		return err
	}
```

The `reconcileContainerState` function compares `desired != actual` to decide whether to stop a container. If `spec.state` is `"created"` (which expresses user intent for the container to be stopped), and the actual state is `"exited"` or `"dead"`, the condition `desired != actual` evaluates to true. The function then unconditionally calls `w.cli.ContainerStop`. Since `ContainerStop` on an already-stopped container natively succeeds and returns `nil`, the reconciler completes successfully without updating any K8s status. Consequently, every subsequent watch event across the entire system (images, volumes, other containers) triggers `reconcileContainerSpecs`, which iterates over all containers and re-issues `ContainerStop` for every stopped container, resulting in an endless O(N) Docker API spam loop.

Fix: Consider the `"created"` desired state satisfied if the container is already in a stopped state.

```diff
 	if desired == actual {
 		return nil
 	}
+
+	// A desired state of "created" expresses intent for the container to be
+	// stopped. If the container is already stopped, this intent is satisfied.
+	if desired == containersv1alpha1.ContainerStatusCreated {
+		switch actual {
+		case containersv1alpha1.ContainerStatusCreated,
+			containersv1alpha1.ContainerStatusExited,
+			containersv1alpha1.ContainerStatusDead:
+			return nil
+		}
+	}
 
 	log := logf.FromContext(ctx).WithName("docker-watcher")
```

#### Important Issues

**I1. CrashLoopBackOff bypassed for `spec.state=running`** ŌĆö `pkg/controllers/app/engine/controllers/docker_watcher.go:288-296` (important, design gap)

When the user requests `spec.state=running` and the container continuously crashes, its status rapidly changes to `exited`. `desired (running) != actual (exited)` triggers an immediate `ContainerStart` call. The container runs, crashes, emits a `die` event, K8s state updates to `exited`, and the resulting watch event triggers another immediate `ContainerStart`. This circumvents Docker's native restart policies and creates an unthrottled restart loop entirely driven by K8s watch events without any backoff mechanism.

Fix: Rely on Docker's native restart policies for robust container lifecycles, and explicitly document that enforcing `spec.state=running` does not provide native `CrashLoopBackOff`. If strict declarative restarts are required, implement an exponential backoff state tracking in the controller before invoking `ContainerStart`.

#### Suggestions

**S1. Missing channel closure check in Docker watcher loop** ŌĆö `pkg/controllers/app/engine/controllers/docker_watcher.go:178-183` (suggestion, robustness)

```go
		case msg := <-result.Messages:
			if err := w.handleEvent(ctx, msg); err != nil {
				log.Error(err, "Failed to handle Docker event",
					"type", msg.Type, "action", msg.Action, "actor", msg.Actor.ID)
			}
```

The `<-result.Messages` select case does not check the `ok` boolean. If the channel is closed (e.g. daemon disconnect), it continuously yields a zero-value message. While `handleEvent` safely ignores empty messages and the loop will eventually terminate when `select` randomly picks the closed `result.Err` case, it's safer to explicitly check for closure to avoid any CPU spin cycles.

Fix: Check `ok` and exit gracefully:

```diff
-		case msg := <-result.Messages:
+		case msg, ok := <-result.Messages:
+			if !ok {
+				log.Info("Docker event stream closed")
+				w.enqueueReconcile()
+				return
+			}
```

**S2. Missing `RetryOnConflict` when stripping finalizers** ŌĆö `pkg/controllers/app/engine/controllers/engine_reconciler.go:375-382` (suggestion, consistency)

```go
		if removeFinalizer(c, mirrorFinalizer) {
			if err := client.IgnoreNotFound(r.Update(ctx, c)); err != nil {
				errs = append(errs, fmt.Errorf("failed to remove finalizer from Container %s: %w", c.Name, err))
			}
		}
```

`processContainerFinalizers` uses a direct `r.Update(ctx, c)` to remove the mirror finalizer. If the object was updated concurrently, this returns a conflict error and relies on the controller-runtime to requeue and retry the entire `deleteContainer` operation. While `deleteContainer` is idempotent, handling conflicts locally using `retry.RetryOnConflict` avoids unnecessary API latency and matches the pattern established in `removeMirrorResource`.

Fix: Wrap the finalizer removal in a `retry.RetryOnConflict` block that re-fetches the container before updating.
