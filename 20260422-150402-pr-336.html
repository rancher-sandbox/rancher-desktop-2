Executive Summary ¶

Round 6 (fca9713) closes every Round 5 Important: watchUntil / watchUntilDeleted now re-Get on watch close and distinguish a satisfied predicate from a dropped watch (cmd/rdd/limavm.go:497-509, 547-563); StopWithWait's deadline branch now splits on ctx.Err(), so context.Canceled is no longer reported as a timeout (pkg/service/cmd/service.go:383-397); and both ensureServiceRunning and serviceStartAction's already-running branches now anchor waitForFreshDiscoveryConfigMap on service.StartTime() (cmd/rdd/service.go:98-110, 262-282). The library-surface refactor — StopWithWait(ctx, wait, timeout), Delete(ctx, timeout), the new StartTime() — and the centralization of context.DeadlineExceeded → exit-4 routing in cliexit.Classify are clean and well-documented.

Two new Importants surface in this round. I1: cliexit.Classify (pkg/cli/exit/exit.go:57-66) short-circuits only when the existing *Error already has Code == CodeTimeout. Any future call site that wraps a deadline-bearing error with a different code (cliexit.Rejected(ctx.Err()), or a not-yet-defined code) will have its classification silently stripped and replaced with Timeout. No current caller produces that combination, so the bug is latent, but the docstring promises "callers may Classify repeatedly" — the fix is a one-line broadening of the short-circuit. I2: StopWithWait deletes the discovery ConfigMap before sending the signal (pkg/service/cmd/service.go:355-356), and on a short timeout process.Kill returns before the process actually exits. A user who runs rdd svc start in that window now takes the already-running branch, calls waitForFreshDiscoveryConfigMap with the current PID-file mtime as anchor, and waits 90s (or until startWaitTimeout) for a ReadyAnnotation that the dying service cannot write. The new test bats/tests/20-service/6-timeout.bats:22-29 exercises this exact sequence and relies on the service exiting fast enough to take the cold-start path instead.

Ten Suggestions follow. S1 captures Codex's observation that the second-precision freshness anchor (PID-file mtime) could theoretically accept a stale ConfigMap when a prior unclean shutdown lands in the same wall-clock second as the new startup — demoted from Important because the pre-PR already-running branch had no freshness check at all, so Round 6 is still strictly better, and the race window is sub-second. S2 and S3 capture Claude's two demoted Importants: watchUntil's watch-close + NotFound fall-through returns a raw apierror instead of the symmetric "deleted while waiting" message, and the Running() → StartTime() call pair has a theoretical ENOENT race that surfaces as an opaque stat … no such file or directory error. Both are one-line fixes. The remaining suggestions cover a missing exit-4 contract test for the shared autostart helpers (svc config, set, ctl), a wait local variable that shadows the k8s.io/apimachinery/pkg/util/wait import, a comment/predicate mismatch in Delete, the BATS setup_rdd_control_plane helper's now-5-minute potential hang, a timing dependency in the new 6-timeout test, and two doc/flag-help polish items.

No Critical issues. Design Observations ¶

Strengths

  • watchtools.ContextWithOptionalTimeout replaces the bespoke time.After channel in StopWithWait, unifying cancellation and timeout on the same primitive and inheriting client-go's "0 = forever" idiom. (in-scope) Claude
  • The watchUntil / watchUntilDeleted fall-through — re-Get after channel close, distinguishing a satisfied predicate from a dropped watch — closes Round 5's silent-success regression. (in-scope) Gemini
  • Centralizing context.DeadlineExceeded → exit 4 in cliexit.Classify keeps exit-code policy at the CLI boundary instead of scattering per-command checks. (in-scope) Codex Gemini
  • Anchoring waitForFreshDiscoveryConfigMap to service.StartTime() in the already-running path closes Round 5's staleness gap and makes the already-running and cold-start paths use the same predicate. (in-scope) Claude
  • Delete(ctx, timeout) preserves instance.Dir() on a timed-out stop, per the documented Windows directory-corruption and Unix PID-file mutual-exclusion hazards. (in-scope) Gemini

Concerns

  • Delete leaks StopWithWait's internal error vocabulary into the caller via errors.Is(err, context.DeadlineExceeded) || Running(). A typed sentinel from StopWithWait — or returning a *cliexit.Error directly — would let Delete express its contract in one check and also retire I1's double-classification risk by construction. (future) Claude
  • Readiness freshness models identity across two storage systems (filesystem mtime and Kubernetes CreationTimestamp). An explicit startup nonce copied from InitDiscovery into the ConfigMap, and matched in waitForFreshDiscoveryConfigMap, would remove clock-precision coupling and let the docs describe readiness identity exactly rather than via timestamp ordering. (future) Codex
  • service.Start(ctx, …) launches the serve subprocess via exec.CommandContext(ctx, …) where ctx is the CLI command context. A SIGINT during a long limavm start --wait would cancel that context and kill the just-started daemon; background daemons typically need context.Background() (or a detached context) so parent cancellation does not tear them down. Unchanged by this PR. (future) Gemini

Repo context updates ¶

  • [repo] ADD a verification heuristic for latent idempotency guarantees in wrappers. When a new helper function's docstring says "callers may X repeatedly" or "X is idempotent," read the implementation against every existing caller that could violate the guarantee, then trace any plausible future caller the signature invites. Latent bugs of this shape rarely surface until the invited caller lands, at which point the behaviour regresses quietly. Flag when the implementation's short-circuit is narrower than the docstring claims. Rationale: the Classify finding this round existed because the docstring promised more than the implementation delivered, and no caller today exercises the gap.
  • [repo] ADD pairwise verification of CLI state cleanup and CLI readiness waits. When a CLI command both (a) deletes shared cluster state during shutdown and (b) another CLI command waits on that state for readiness, trace the sequence shutdown-with-timeout → state-deleted → restart-quick → readiness-wait-against-missing-state. This pattern produces user-visible hangs the individual code paths look clean. Rationale: I2 this round is exactly this shape — the stop deletes the discovery CM, the start waits for the same CM with a freshness anchor, and the window between signal send and actual process exit is the failure mode.
