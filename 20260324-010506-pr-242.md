## Executive Summary

This PR migrates Windows CI from WSL2-based BATS to MSYS2, fixes Windows process signaling so hostagent shutdown no longer kills the RDD service, adds WSL2 distro lifecycle management to prevent `wslservice.exe` deadlocks, and pre-generates SSH keys to avoid MSYS2 path conversion issues. The code is well-structured with thorough comments explaining platform-specific behavior. One important regression exists in the SSH key recovery path where a transient `ssh-keygen` failure deletes the only copy of the private key, breaking SSH access to existing VMs.

## Important Issues

1. **SSH key recovery deletes valid private key on transient ssh-keygen failure** тАФ `pkg/controllers/lima/limavm/controllers/ssh_keys.go:37-40` [Codex GPT 5.4] (important, regression)

```go
// Public key missing тАФ try to derive it from the existing private key
// to preserve SSH access to existing VMs. Fall back to full regeneration
// if the private key is corrupt.
if pub, err := exec.CommandContext(ctx, "ssh-keygen", "-y", "-f", privPath).Output(); err == nil {
	return os.WriteFile(pubPath, pub, 0o644)
}
_ = os.Remove(privPath)
```

When the public key is missing but the private key is valid, `ensureSSHKeysAt` tries to derive the public key via `ssh-keygen -y` (line 37). If that command fails for *any* reason тАФ `ssh-keygen` missing from `PATH`, a transient execution error, or the 30-second context timeout тАФ the code deletes the valid private key at line 40 before falling through to full regeneration. Because `SetupWithManager()` calls `ensureSSHKeys()` on every controller startup (line 453 of `limavm_controller.go`), a single transient failure destroys the only identity that existing VMs trust.

The regeneration at lines 52тАУ74 creates a fresh keypair, but VMs provisioned with the old key become unreachable. In this pre-release project VMs can be recreated, which limits the blast radius, but the fix is straightforward.

Fix: Do not delete the private key until a replacement is confirmed in place. Remove the eager `os.Remove(privPath)` at line 40, and let the rename sequence at lines 65тАУ74 atomically replace the old key only after generation succeeds.

```diff
 	if pub, err := exec.CommandContext(ctx, "ssh-keygen", "-y", "-f", privPath).Output(); err == nil {
 		return os.WriteFile(pubPath, pub, 0o644)
 	}
-	_ = os.Remove(privPath)
 }
```

The generate-to-temp-then-rename pattern already handles the case where `privPath` exists on Windows (line 44 removes the orphaned pub key, and the rename at line 70 overwrites the private key only after the new one is ready). The only additional consideration: on Windows `os.Rename` fails if the destination exists, so add an `os.Remove(privPath)` immediately before the rename at line 70, matching the pattern already used for `pubPath` at line 44.

## Design Observations

### Concerns

**Stale `inst` in `shutdownHostagent` forceStop closure** тАФ (future) [Claude Opus 4.6, Gemini 3.1 Pro]

When `shutdownHostagent` receives a non-nil `inst` (e.g., from `stopInstance` at line 478), the `forceStop` closure captures that pointer at line 521. After the 30-second graceful timeout, the hostagent's PID may have been freed and recycled (especially on Windows). `stopInstanceForcibly` then kills by PID without identity verification.

Gemini proposed a fresh `store.Inspect` in the force path as a fix. However, `store.Inspect` reads from the same on-disk PID files the hostagent wrote. If the hostagent crashed without cleaning up (the reason we're in the force path), `store.Inspect` returns the same stale PIDs. The fresh inspect helps only in a narrow race window where the hostagent exits cleanly between the timeout check and the force-kill call.

The code already acknowledges this at lines 215тАУ224 and identifies the real fix: process identity validation or Windows Job Objects. The self-healing restart limits blast radius.

### Strengths

- Replacing Lima's `SysKill` (which broadcasts `CTRL_BREAK_EVENT` to the entire console) with targeted `process.Interrupt` + `process.KillTree` eliminates the root cause of hostagent shutdown killing the RDD service.
- The SSH key pre-generation in `ensureSSHKeysAt` handles every partial-state combination with clean error recovery. The generate-to-temp-then-rename pattern prevents blocked retries from partial keys.
- `stopInstanceForcibly` conditions disk cleanup (lock and tmp file removal) on `allKilled` success, preventing status desynchronization where removing PID files while the process is still alive would make `store.Inspect` falsely report `StatusStopped`.
- The WSL2 distro termination before `--unregister` with the BATS-level `taskkill /F /IM wslservice.exe` fallback is a practical defense against a kernel-level deadlock that cannot be fixed from userspace.
- `setVerbosityFromEnv` cleanly bridges the service's logrus string log levels to the external controllers' klog numeric flags without adding abstraction overhead.

**Stale `inst` in `shutdownHostagent` forceStop closure** тАФ (future). ### Gemini 3.1 Pro Review

### Executive Summary

This PR successfully migrates Windows BATS testing from WSL2 interop to native MSYS2, mitigating flaky test runs caused by GitHub Actions runners. It isolates the RDD service and Lima hostagents into separate process groups to prevent RDD from crashing when VMs restart, and it hardens WSL2 instance lifecycle handling against deadlocks. However, an important regression was introduced during forced instance termination where stale PIDs can result in arbitrary host processes being killed on Windows due to aggressive PID recycling.

### Critical Issues

None.

### Important Issues

1. **Stale instance state used for forced termination after graceful timeout** тАФ `pkg/controllers/lima/limavm/controllers/limavm_lifecycle.go:514-528` and `569-587` (important, regression)

```go

		// Use a background context: the parent reconciler context may be
		// nearing its deadline after the graceful shutdown wait.
		forceCtx, forceCancel := context.WithTimeout(context.Background(), 30*time.Second)
		defer forceCancel()
		forceInst := inst
		if forceInst == nil {
			var err error
			forceInst, err = store.Inspect(forceCtx, name)
```

In `shutdownHostagent` and `killOrphanedHostagent`, the `inst` parameter passed from the caller is often captured *before* the 30-second graceful shutdown wait (e.g., at line 478 in `stopInstance`). If the wait times out, or if the process exits right around the timeout boundary, the PID contained in `inst.HostAgentPID` will be freed and potentially recycled by Windows. `forceStop` then blindly passes this stale PID to `stopInstanceForcibly`, which executes `taskkill /F /T /PID`, risking termination of an unrelated process tree. The `shutdownAllHostagents` function properly avoids this by performing a fresh `store.Inspect` *after* the wait, but these paths do not.

-			}

-				return
