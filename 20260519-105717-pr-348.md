## Important Issues

I1. **`logrus.Warnf` in routine cache-miss download surfaces a WARN banner at default log level** — `pkg/kuberlr/downloader.go:78` [Claude Opus 4.7, Gemini 3.1 Pro]

```go
logrus.Warnf("downloading kubectl v%d.%d.%d from %s", want.Major, want.Minor, want.Patch, mirrorURL())
```

Outside developer mode, `RDD_LOG_LEVEL` defaults to `warn`, so every `logrus.Warn` reaches the user's terminal. A cache miss on a remote cluster's first invocation at a new minor version is a routine expected event: the resolver downloads, sha-verifies, and execs. The user sees a yellow `WARN[…]` banner above their normal `kubectl get pods` output. Compare to the `logrus.Debug` lines elsewhere in the resolver for routine fall-through paths.

The intent is legitimate — telling the user "you are about to wait for a ~50 MB download" is operator-relevant — so the choice of level is a judgment call. Claude raised it as such; Gemini flagged it as Warn-level misuse outright. Two reasonable shapes for the fix:

```diff
- logrus.Warnf("downloading kubectl v%d.%d.%d from %s", want.Major, want.Minor, want.Patch, mirrorURL())
+ // One-time per (server-version, cache state) event. Surface what the
+ // resolver is doing without raising a WARN; the user sees kubectl
+ // output immediately after.
+ logrus.Infof("downloading kubectl v%d.%d.%d from %s", want.Major, want.Minor, want.Patch, mirrorURL())
```

Or keep `Warnf` and document the rationale inline against the project's logging convention.

I2. **`local_teardown_file()` cannot terminate the native Win32 fake-kube-server.exe** — `bats/tests/10-cli/7-kubectl-cache.bats:91-99` [Codex GPT 5.5]

```bash
local_teardown_file() {
    if load_var SERVER_PID; then
        kill "${SERVER_PID}" 2>/dev/null || true
    fi
}
```

`local_setup_file()` builds the fake server as `fake-kube-server.exe` on Windows and launches it (lines 35-39, 49-55). MSYS2 `kill` cannot reach Win32 processes — `bats/helpers/os.bash:96-117` documents this case explicitly and provides `process_kill`, which dispatches to `taskkill.exe /F /PID` on Windows. The existing teardown silently fails on MSYS, leaking the HTTP server and the bound ephemeral port across BATS sessions, which is the exact behavior the deviation from the "no `teardown_file`" rule was added to prevent. The cleanup works on macOS and Linux.

```diff
 local_teardown_file() {
     if load_var SERVER_PID; then
-        kill "${SERVER_PID}" 2>/dev/null || true
+        process_kill "${SERVER_PID}" 2>/dev/null || true
     fi
 }
```

I3. **`run -0 env … rdd …` bypasses the `rdd()` shell function on WSL** — `bats/tests/10-cli/5-paths.bats:45,48`; `bats/tests/10-cli/7-kubectl-cache.bats:124-128,143-147,159-163,178-182,199-203` [Codex GPT 5.5]

```bash
run -0 env \
    RDD_CACHE_DIR="$(winpath "${CACHE_DIR}")" \
    RDD_KUBECTL_MIRROR="http://127.0.0.1:${PORT}" \
    KUBECONFIG="$(winpath "${KUBECONFIG_PATH}")" \
    rdd kubectl get pods
```

`bats/helpers/commands.bash:42-89` defines `rdd()` as a bash function that handles three Windows boundaries: MSYS path conversion (lines 47-61), WSL `/mnt/...` path conversion (lines 62-70), and the WSLENV augmentation (lines 71-82) that adds every `RDD_*` env var to WSLENV so they cross the WSL→Win32 boundary into `rdd.exe`. `bats/helpers/load.bash:65` puts `bin/` on PATH, so `env … rdd …` execvp's `bin/rdd${EXE}` directly — none of the wrapper logic runs. Two consequences on WSL:

```bash
# bats/helpers/commands.bash:71-82 — WSLENV augmentation
WSLENV=""
for env in "${envs[@]}"; do
    WSLENV="${WSLENV}:${env}"
done
```

First, `RDD_CACHE_DIR` and `RDD_KUBECTL_MIRROR` set on the `env` command line do not get added to WSLENV, so the values do not cross into `rdd.exe`. Second, `winpath` (see S1) does not handle WSL, so the pre-conversion in the test produces Linux-form paths that `rdd.exe` cannot read.

The fix routes the call through the bash function. Two ways:

```diff
-    run -0 env \
-        RDD_CACHE_DIR="$(winpath "${CACHE_DIR}")" \
-        RDD_KUBECTL_MIRROR="http://127.0.0.1:${PORT}" \
-        KUBECONFIG="$(winpath "${KUBECONFIG_PATH}")" \
-        rdd kubectl get pods
+    RDD_CACHE_DIR="${CACHE_DIR}" \
+    RDD_KUBECTL_MIRROR="http://127.0.0.1:${PORT}" \
+    KUBECONFIG="${KUBECONFIG_PATH}" \
+    run -0 rdd kubectl get pods
```

The temporary-assignment-before-command form invokes the `rdd` function with the env vars set for that one call; the function then handles path conversion and WSLENV. (`winpath` becomes redundant once the wrapper handles it.)

## Suggestions

S1. **`winpath` does not handle WSL→Windows.exe path conversion** — `bats/helpers/paths.bash:14-20` [Codex GPT 5.5]

```bash
winpath() {
    if is_msys; then
        cygpath -w "$1"
    else
        printf '%s' "$1"
    fi
}
```

`winpath` only converts on MSYS. WSL is the other shell that calls a native Windows `rdd.exe` (`commands.bash:4-6` sets `EXE=.exe` whenever `using_windows_exe` is true), and the existing repo convention for WSL path conversion is `wslpath -w` (see `commands.bash:65-66` and `paths.bash:26-27`). On WSL the `$(winpath ...)` call sites pass Linux-form paths into `rdd.exe`, which cannot read them. The narrow fix:

```diff
 winpath() {
     if is_msys; then
         cygpath -w "$1"
+    elif is_windows && using_windows_exe; then
+        wslpath -w "$1"
     else
         printf '%s' "$1"
     fi
 }
```

A wider fix routes the call sites through the `rdd()` shell function (see I3); doing both is defensible because `winpath` is also used in `local_setup_file` for raw `${SERVER_BIN}` args that never traverse the `rdd()` wrapper.

S2. **Unbound kubectl-global bool flags can hide the subcommand from `isClientOnly`** — `pkg/kuberlr/resolver.go:230-264` [Claude Opus 4.7]

```go
var help, client, warningsAsErrors bool
fs.BoolVarP(&help, "help", "h", false, "")
fs.BoolVar(&client, "client", false, "")
fs.BoolVar(&warningsAsErrors, "warnings-as-errors", false, "")

The fall-through is correct (no silent mismatched execution), but the user pays the 2-second probe penalty. `--match-server-version` is the obvious candidate to add; an audit of upstream `staging/src/k8s.io/kubectl/pkg/cmd/cmd.go` would surface any other bool globals kubectl declares above the subcommand layer.

```diff

+var matchServerVersion bool

A mirror of the existing `--warnings-as-errors` test cases locks the behavior in: `{"--match-server-version before config", []string{"--match-server-version", "config", "view"}, true}`.

S3. **`skipResolver` global is a set-only toggle that leaks across hypothetical future tests** — `pkg/kuberlr/resolver.go:31-39`, `pkg/kuberlr/resolver_test.go:222-233` [Claude Opus 4.7]

```go

S4. **`main.go` `errors.As` block captures `msg` it does not use; check `exitErr.Err` directly** — `cmd/rdd/main.go:172-178` [Claude Opus 4.7]

```go
if errors.As(err, &exitErr) {
    if msg := err.Error(); msg != "" {
        logrus.Error(err)
    }
    os.Exit(exitErr.Code)
}
```

The captured `msg` is unused; `logrus.Error(err)` calls `err.Error()` again. The empty-string check works today because `cliexit.Error.Error()` returns `""` iff `Err == nil`. If a future caller wraps the `cliexit.Error` (e.g. `fmt.Errorf("foo: %w", &cliexit.Error{Code: c})`), the outer `err.Error()` is non-empty even when the inner sentinel marks "subcommand already wrote output," and the redundant log line returns. Checking `exitErr.Err != nil` states intent precisely and survives the wrap.

```diff
 if errors.As(err, &exitErr) {
-    if msg := err.Error(); msg != "" {
+    if exitErr.Err != nil {
         logrus.Error(err)
     }
     os.Exit(exitErr.Code)
 }
```

S5. **`winpath` via `$(...)` swallows non-zero exits in standalone assignments** — `bats/tests/10-cli/5-paths.bats:43`; `bats/tests/10-cli/7-kubectl-cache.bats:39` [Claude Opus 4.7]

```bash
cache_override=$(winpath "${BATS_TEST_TMPDIR}/cache-override")
```

The repo's BATS style rule prefers `run -0 cmd ...; value=${output}` over `value=$(cmd ...)` because command substitution discards the exit code and leaves the variable empty on failure. `winpath` delegates to `cygpath -w` on MSYS, which can fail in pathological cases. At the standalone assignment sites — `5-paths.bats:43` and the `SERVER_BIN=...` shape near `7-kubectl-cache.bats:39` — the `run -0` form locks in the safety. Sites inside a `run` command line (most of `7-kubectl-cache.bats`) cannot use `run -0` recursively; those are unavoidable shell composition.

S6. **BATS coverage forces every test out of skew; the within-skew "no download" path runs only in unit tests** — `bats/tests/10-cli/7-kubectl-cache.bats` [Claude Opus 4.7]

The five `7-kubectl-cache.bats` tests all pin the fake apiserver at `v1.99.0`, always out of skew with the embedded kubectl. The end-to-end coverage exercises the download lifecycle (miss, hit, mirror 404, sha mismatch, apiserver 500 fall-through) but never the within-skew branch where `compatible()` returns true and `Resolve` returns `""` without probing the mirror. The within-skew rule is covered by `TestCompatible` at the unit level; an end-to-end test (e.g. set `--git-version` to `v$(EMBEDDED_MINOR).0`) would prove the whole resolver respects the rule end-to-end.

S7. **Per-patch cache key accumulates one binary per patch release** — `pkg/kuberlr/downloader.go:86-92` [Claude Opus 4.7] (future)

```go
name := fmt.Sprintf("kubectl-v%d.%d.%d", v.Major, v.Minor, v.Patch)
```

A cluster stepping through `v1.30.5 → v1.30.6 → v1.30.7` accumulates three ~50 MB binaries even though any one would satisfy the skew rule. The eviction TODO in `cache.go:33-39` addresses growth but not consolidation. A future enhancement could cache one binary per server minor (the highest patch the mirror knows about for that minor) and reuse it for any server in the same minor band. ## Design Observations

### Concerns

- **Double execution of KUBECONFIG `exec` auth plugins (in-scope)** [carried over from Round 7 retro]. `serverVersion` calls `discovery.NewDiscoveryClientForConfig(cfg).ServerVersion()`, which runs the user's kubeconfig `exec` credential plugin (SSO, IDP CLI, hardware-token). The probe runs the plugin once; if Resolve then downloads and execs a remote kubectl, kubectl invokes the same plugin again. Non-interactive plugins double the latency; interactive plugins double the prompts. The TODO at `resolver.go:64-67` for `RDD_KUBECTL_OFFLINE` is the natural escape hatch — offline mode should also suppress the probe so the user's `exec` plugin runs exactly once.

### Strengths

- **`genericclioptions.ConfigFlags` for resolver flag binding** [Codex, Gemini]. The probe and the kubectl child stay aligned on kubeconfig precedence by construction — any future kubectl flag rename or addition flows through `NewConfigFlags(true).AddFlags(fs)` without a parallel implementation to drift.
- **Body-size caps and per-request timeout on `streamGet`** [Claude]. `maxSha512Bytes` (4 KiB) and `maxKubectlBytes` (256 MiB), paired with `downloadTimeout` (5 min), turn a malicious or misconfigured mirror into a bounded failure rather than an unbounded write.
- **Sibling Go module for fake-kube fixtures** [Claude, Gemini]. `bats/tests/10-cli/fake-kube/go.mod` keeps test-only dependencies out of the parent module, and a single server plays both apiserver `/version` and release mirror `/release/*` roles with file-based per-test overrides.
- **Conservative probe fallback policy** [Codex, Gemini]. Missing kubeconfig, malformed kubeconfig, unreachable cluster, and unparseable server version all fall through silently to the embedded kubectl. Only download and sha errors surface loudly, matching the "user wants to know" cut explicitly drawn in the resolver's comments.
- **Cross-process recursion guard** [Claude]. The `RDD_KUBECTL_RESOLVED` env var plus the same-process `skipResolver` flag together cover both axes of recursion: kubectl reentry through a shim, and `rdd ctl` short-circuiting before it dispatches to `kubectlAction`.

# Code Review — PR #348: rdd kubectl version resolver

## Executive Summary

This PR adds a kuberlr-style version resolver to `rdd kubectl`: when the user's cluster is more than ±1 minor out of skew with the embedded kubectl, rdd probes `/version`, downloads a matching kubectl from `dl.k8s.io` (sha512-verified) into a shared `RDD_CACHE_DIR` cache, and exec's it. The design is well documented and the failure paths are deliberately chosen. Findings cluster around a UX-regression Warn at the default log level, fragility in the package-global `skipResolver` flag, and a small set of kubectl global flags the resolver's flag binding does not cover.

**S2. Concurrent `rdd kubectl` cache-fill race on Windows** — `pkg/kuberlr/downloader.go:68-129`

```go
func ensureCached(ctx context.Context, want semver.Version) (string, error) {
    path := cachedPath(want)
    if _, err := os.Stat(path); err == nil {
        return path, nil
    }
    ...
    return os.Rename(tmpName, dst)
}
```

Two concurrent invocations (different terminals, same user) with an empty cache both stat → ENOENT, both download to distinct temp names, both verify, both `os.Rename` to the same `dst`. On Unix the last writer wins and both processes hold the inode they exec'd, so both work. On Windows, if process A has already exec'd `dst` via `CreateProcess`, the OS holds an exclusive lock on the file and B's `os.Rename` fails with `ERROR_SHARING_VIOLATION`, surfacing as `resolving kubectl version: rename ...`. A user retry succeeds because the cache now contains the binary.

This is rare and self-heals on retry, so it is SUGGESTION severity. But the failure mode is operator-confusing ("rename" error on a download path is hard to map back to the actual issue).

Fix: on rename failure, re-stat the destination and accept it if it now exists with a matching sha512.

```diff
- if err := os.Chmod(tmpName, 0o755); err != nil {
-     return err
- }
- return os.Rename(tmpName, dst)
+ if err := os.Chmod(tmpName, 0o755); err != nil {
+     return err
+ }
+ if err := os.Rename(tmpName, dst); err != nil {
+     // Concurrent install? Accept the racer's binary if it landed.
+     if _, statErr := os.Stat(dst); statErr == nil {
+         return nil
+     }
+     return err
+ }
+ return nil
```

### Executive Summary

This PR adds a kuberlr-style resolver that probes the target cluster, downloads a version-matched kubectl into a shared cache when needed, and exposes the cache paths. The main product issue is a Windows cache publication race; the test suite also has Windows cleanup/path portability problems.

#### Important Issues

I1. **Concurrent cache misses can fail on Windows** — `pkg/kuberlr/downloader.go:68-80,128` (important, regression)

```go
if _, err := os.Stat(path); err == nil {
	return path, nil
}
...
if err := download(ctx, want, path); err != nil {
	return "", err
}
```

```go
return os.Rename(tmpName, dst)
```

The cache is shared across instances, but `ensureCached` checks for absence at line 70 and then downloads without any publication guard at line 79. If two `rdd kubectl` processes miss the same version concurrently, both can validate a temp file; the first `os.Rename` at line 128 publishes the cache entry, and the second fails on Windows because `os.Rename` does not replace an existing destination there. The second command then exits with `resolving kubectl version` even though a valid cache entry now exists.

Fix: either serialize per-version downloads with a lock file, or make publication tolerate a winner. If handling `os.IsExist`, re-hash the now-existing destination against `want` before returning success so a stale/corrupt file is not accepted blindly.

```diff
	if err := os.Chmod(tmpName, 0o755); err != nil {
		return err
	}
-	return os.Rename(tmpName, dst)
+	if err := os.Rename(tmpName, dst); err != nil {
+		if os.IsExist(err) {
+			return verifyCachedFile(dst, want)
+		}
+		return err
+	}
+	return nil
```

### Executive Summary

This PR integrates a `kuberlr`-style version resolver into the `rdd kubectl` command. It correctly proxies invocations, parses user connection overrides, probes the target cluster's version, and falls back to a transparent download/cache layer when the embedded binary is out of skew. The implementation is robust, with excellent test coverage via the `fake-kube` BATS fixtures. A few important issues were found regarding a Windows-specific file collision race condition, a strict panic on missing cache directories, and an inappropriate `Warnf` log level for routine downloads.

### Findings

#### Critical Issues

None.

#### Important Issues

I1. **Windows `os.Rename` race condition in concurrent downloads** — `pkg/kuberlr/downloader.go:126-129` (important, gap)

```go
	if err := os.Chmod(tmpName, 0o755); err != nil {
		return err
	}
	return os.Rename(tmpName, dst)
```

On Windows, `os.Rename` fails with `ERROR_ALREADY_EXISTS` (mapped to `os.ErrExist`) if the destination file already exists, unlike Unix which replaces it atomically. If two `rdd kubectl` invocations hit a cache miss for the same version concurrently, they will both download to unique `.kubectl-*` temp files and both attempt to rename them to `dst`. The first process succeeds, but the second fails and returns an error, aborting the user's `kubectl` invocation even though a valid binary is now successfully cached.

Fix: Catch the error on Windows and consider the operation successful if the destination file exists.

```diff
 	if err := os.Chmod(tmpName, 0o755); err != nil {
 		return err
 	}
-	return os.Rename(tmpName, dst)
+	err := os.Rename(tmpName, dst)
+	if err != nil && runtime.GOOS == "windows" {
+		if _, statErr := os.Stat(dst); statErr == nil {
+			return nil
+		}
+	}
+	return err
 }
```

I2. **Panic on missing user cache directory** — `pkg/kuberlr/cache.go:48-51` (important, gap)

```go
	cache, err := os.UserCacheDir()
	if err != nil {
		panic(fmt.Errorf("could not determine user cache directory: %w", err))
	}
```

`CacheRoot()` evaluates `os.UserCacheDir()` and panics if it fails (e.g., if `$HOME` is unset in a constrained CI/Docker environment). Because `instancePaths()` in `cmd/rdd/service_paths.go` evaluates `kuberlr.CacheRoot()` unconditionally, running `rdd svc paths` in such an environment will crash the entire `rdd` CLI process. While a missing cache directory might impede binary downloads, it should not crash informational CLI commands.

Fix: Provide a graceful fallback to the OS temporary directory rather than panicking.

```diff
 	cache, err := os.UserCacheDir()
 	if err != nil {
-		panic(fmt.Errorf("could not determine user cache directory: %w", err))
+		cache = os.TempDir()
 	}
 	return filepath.Join(cache, "rancher-desktop")
```

I3. **Misuse of Warn level for routine informational output** — `pkg/kuberlr/downloader.go:78-80` (important, regression)

```go
	if err := os.MkdirAll(filepath.Dir(path), 0o755); err != nil {
		return "", err
	}
	logrus.Warnf("downloading kubectl v%d.%d.%d from %s", want.Major, want.Minor, want.Patch, mirrorURL())
	if err := download(ctx, want, path); err != nil {
```

A cache miss on a new cluster version is a routine, expected path. Logging the download start at `Warn` level breaks standard logging semantics and surfaces a yellow/red warning to the user for a successful operation. While this bypasses the default `--log-level=warn` filter to keep the user informed, warning channels should be reserved for degraded or unexpected conditions.

Fix: Standard CLI tools print routine status updates directly to standard error, leaving structured logs for actual warnings.

```diff
 	if err := os.MkdirAll(filepath.Dir(path), 0o755); err != nil {
 		return "", err
 	}
-	logrus.Warnf("downloading kubectl v%d.%d.%d from %s", want.Major, want.Minor, want.Patch, mirrorURL())
+	fmt.Fprintf(os.Stderr, "downloading kubectl v%d.%d.%d from %s\n", want.Major, want.Minor, want.Patch, mirrorURL())
 	if err := download(ctx, want, path); err != nil {
```

**Strengths**

### Acknowledged Limitations

- **Temp file leak on SIGKILL**: `pkg/kuberlr/cache.go:34` — "SIGKILL or power loss during a download also leaks the .kubectl-* temp file..." The author explicitly recognizes the lack of a garbage collection loop.
- **Offline download fallback**: `pkg/kuberlr/resolver.go:64` — "TODO(offline): when no cached kubectl matches and the network is down... Add an opt-out for download attempts..." The feature currently fails instead of finding the next best cached matching version.
- **Windows Ctrl+C propagation**: `pkg/kuberlr/exec_windows.go:20` — "CommandContext hard-kills on ctx cancellation... rdd's hard-kill races kubectl's graceful shutdown of the console-delivered Ctrl+C." The author documented that graceful signal handling is deferred.
- **Max file size truncation limits**: `pkg/kuberlr/downloader.go:142` — "If kubectl ever exceeds maxKubectlBytes, io.LimitReader truncates silently and the digest covers a partial body..." The author notes it will safely fail checksum verification rather than logging a dedicated size-limit warning.
- **Malformed boolean flags in client-only check**: `pkg/kuberlr/resolver.go:197` — "Bool-parse errors short-circuit pflag's parser... [and] lands in the empty-positionals branch below (return true)." The codebase notes that `kubectl` itself will reject the bad flag later, avoiding any unverified network request.
- **BATS custom teardown usage**: `bats/tests/10-cli/7-kubectl-cache.bats:90` — "Deviates from the 'no teardown_file' rule. The rule's intent is to preserve rdd state for post-mortem inspection..." The local ephemeral server runs as a background process and must be cleaned up to prevent leaking HTTP ports across BATS sessions.
