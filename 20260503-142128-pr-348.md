S2. **BATS test captures `version_requests` via command substitution where the project rule mandates `run -0`** — `bats/tests/10-cli/7-kubectl-cache.bats:216` [Codex GPT 5.5][Gemini 3.1 Pro]

```bash
version_requests=$(awk '/^GET \/version$/ {n++} END {print n+0}' "${LOG_FILE}")
[[ ${version_requests} -ge 2 ]] ||
    fail "expected >= 2 GET /version (resolver + embedded kubectl), got ${version_requests}: $(cat "${LOG_FILE}")"
```

The project's BATS style (`deep-review-context.md` and `bats-style.md`) explicitly forbids command substitution for command output: "Use `run -0` instead of command substitution. `run -0` fails with a useful error (command, exit code, output) if the command returns non-zero. A bare substitution gives an empty variable and a cryptic failure later." The new assertion uses both the forbidden capture form and a second `$(cat …)` inside the `fail` argument.

Fix:

```diff
-    version_requests=$(awk '/^GET \/version$/ {n++} END {print n+0}' "${LOG_FILE}")
-    [[ ${version_requests} -ge 2 ]] ||
-        fail "expected >= 2 GET /version (resolver + embedded kubectl), got ${version_requests}: $(cat "${LOG_FILE}")"
+    run -0 awk '/^GET \/version$/ {n++} END {print n+0}' "${LOG_FILE}"
+    version_requests=${output}
+    if [[ ${version_requests} -lt 2 ]]; then
+        run -0 cat "${LOG_FILE}"
+        fail "expected >= 2 GET /version (resolver + embedded kubectl), got ${version_requests}: ${output}"
+    fi
```

S4. **`fetchSha512` accepts uppercase hex but never normalises it; the case-sensitive `if got != want` rejects a valid digest from an uppercase-emitting mirror** — `pkg/kuberlr/downloader.go:138-156`, `pkg/kuberlr/downloader.go:110-112` [Codex GPT 5.5]

```go
for _, c := range digest {
    if !(c >= '0' && c <= '9' || c >= 'a' && c <= 'f' || c >= 'A' && c <= 'F') {
        return "", fmt.Errorf("malformed checksum from %s: non-hex character %q", url, c)
    }
}
return digest, nil
```

`fetchSha512` admits `A-F` in the hex validator and returns the raw digest. `download` computes `got := hex.EncodeToString(h.Sum(nil))` (lowercase by Go's spec) and compares with `if got != want`. A mirror populated by PowerShell `Get-FileHash` or any tool emitting uppercase hex passes validation, then surfaces a misleading "checksum mismatch" — exactly the misdirection the digest-format check was added to prevent.

Two coherent options: (a) reject uppercase in the validator (mirror operators normalise), or (b) accept and normalise in `fetchSha512`. (b) is the user-friendlier choice and a one-line edit.

```diff
-    digest := fields[0]
+    digest := strings.ToLower(fields[0])
```

The 128-hex length and character checks then run against the normalised form.

S5. **`isClientOnly` walks past `--as-user-extra` without consuming its value, so the spaced form mistakes `k=v` for the subcommand and triggers an unwanted probe** — `pkg/kuberlr/resolver.go:195-227`, `pkg/kuberlr/resolver.go:160-162` [Claude Opus 4.7]

```go
"--as":                    true,
"--as-group":              true,
"--as-uid":                true,
// --as-user-extra is absent
```

`genericclioptions.NewConfigFlags(true).AddFlags` binds `--as-user-extra` (a `StringArrayVar`, spaced form takes the next arg). The docstring at lines 160–162 names it explicitly:

```
// --insecure-skip-tls-verify, --request-timeout, --as, --as-group,
// --as-uid, --as-user-extra, --cache-dir, --disable-compression,
```

But the value-taking map omits it. `isClientOnly` walks args manually; `--as-user-extra k=v help` lands at `arg = "--as-user-extra"`, the map lookup misses, the walker advances by one (no value consumed), the next arg `k=v` becomes the subcommand candidate, and `clientOnlySubcommands["k=v"]` returns false — so the resolver probes the cluster for an invocation that should have short-circuited.

Net effect: a help / completion / config invocation that uses `--as-user-extra` in spaced form pays the 2 s probe cost. No correctness break (the kubectl child still receives the args correctly), but it regresses the Round 2 `isClientOnly` work for one flag.

Fix: add the entry.

```diff
     "--as-group":              true,
     "--as-uid":                true,
+    "--as-user-extra":         true,
     "--profile":               true,
```

The longer-term option is to drive `isClientOnly` from the same pflag set `parseKubeConfigFlags` already builds; see Design Observations.

S6. **`Resolve` accepts `ctx` but `serverVersion` ignores it; Ctrl+C during the discovery probe waits up to `serverProbeTimeout` (2 s) instead of cancelling immediately** — `pkg/kuberlr/resolver.go:65-93,121-144` [Claude Opus 4.7]

```go
func Resolve(ctx context.Context, args []string) (string, error) {
    ...
    server, ok, err := serverVersion(args)   // ctx dropped
    ...
    return ensureCached(ctx, server)         // ctx forwarded
}

func serverVersion(args []string) (_ semver.Version, ok bool, _ error) {
    cfg, err := loadClientConfig(args)
    ...
    cfg.Timeout = serverProbeTimeout
    info, err := client.ServerVersion()      // no ctx
}
```

`Resolve` forwards `ctx` only to `ensureCached`. The discovery client at line 133 uses `cfg.Timeout` to cap each request at 2 s, but does not honour `ctx.Done()`. Cancellation does not propagate while the probe is in flight; a user pressing Ctrl+C waits up to 2 s per kubectl invocation.

Bounded UX issue, not a correctness issue (the embedded kubectl runs after the probe times out either way). Low priority because the `ensureCached` path already honours `ctx` and the 2 s bound puts a ceiling on the wait.

Fix: thread `ctx` through and bind it to the request. The discovery client exposes the underlying REST client:

```diff
-func serverVersion(args []string) (_ semver.Version, ok bool, _ error) {
+func serverVersion(ctx context.Context, args []string) (_ semver.Version, ok bool, _ error) {
     cfg, err := loadClientConfig(args)
     ...
-    info, err := client.ServerVersion()
+    body, err := client.RESTClient().Get().AbsPath("/version").DoRaw(ctx)
+    var info apimachineryversion.Info
+    if err == nil { err = json.Unmarshal(body, &info) }
```

S8. **`TestParseKubeConfigFlags` row for `--insecure-skip-tls-verify=false` does not exercise the binding; the assertion matches the pre-parse default** — `pkg/kuberlr/resolver_test.go:121` [Claude Opus 4.7]

```go
{"--insecure-skip-tls-verify=false", []string{"--insecure-skip-tls-verify=false", "get"}, want{insecure: false}},
```

`NewConfigFlags(true)` initialises `Insecure` to `*bool → false`. The test asserts the post-parse value equals `false` — the default. The row passes even when the flag binding is absent.

Two fixes: drop the row (the bare and `=true` rows already cover the binding), or force a non-default starting state and assert the parser flipped it back.

```go

*cf.Insecure = true                  // would be overwritten by re-parse; not viable for table tests
```

The table-test shape favours dropping the row.

S9. **Aborted downloads leak `.kubectl-*` temp files in `CacheDir()`; `defer os.Remove` does not cover SIGKILL or hardware crash** — `pkg/kuberlr/downloader.go:96-101` [Gemini 3.1 Pro]

```go
tmp, err := os.CreateTemp(filepath.Dir(dst), ".kubectl-*")
if err != nil {
    return err
}
tmpName := tmp.Name()
defer os.Remove(tmpName)
```

Normal failure paths and orderly cancellation hit the `defer`; SIGKILL during a slow download or a sudden power loss leave a ~50 MB temp file behind. Each subsequent failure adds another. The eviction TODO in `cache.go:33-36` does not currently target temp leftovers either.

Fix: sweep `*.kubectl-*` from `CacheDir()` once at the start of `ensureCached`, or fold a sweep into the eventual eviction pass. Either fix is small enough to land alongside the eviction work; flag here so it is not forgotten.

S10. **`httpClient.Timeout = 5 minutes` covers headers, body, and redirects together; on a 100 kB/s link the body alone exceeds the cap** — `pkg/kuberlr/downloader.go:33-42` [Claude Opus 4.7]

```go
const downloadTimeout = 5 * time.Minute
var httpClient = &http.Client{Timeout: downloadTimeout}
```

The comment estimates "covers a ~50 MB kubectl binary on slow links (~170 kB/s)" — exactly five minutes at that rate, leaving zero headroom for TLS handshake or initial latency, and failing outright below that throughput. Round 3 added the deadline-composition note (a backstop). Round 4 raises the magnitude question: 5 minutes assumes a fast-enough link, and the user on a slow link sees a hard "context deadline exceeded" with no obvious diagnosis.

Two coherent options: (a) bump `downloadTimeout` to 10 minutes (covers ~85 kB/s links), or (b) move from `Client.Timeout` to a transport with `ResponseHeaderTimeout` so only stalls (not slow throughput) trip the cap. (b) is the durable fix; (a) is a one-character edit.

## Design Observations

### Concerns

- **Hand-rolled subcommand walker duplicates the bound pflag set (future)** [Claude Opus 4.7]. `parseKubeConfigFlags` already binds the kubectl global flags via `genericclioptions.ConfigFlags.AddFlags`; after `fs.Parse(args)`, `fs.Args()` returns the positional tokens whose first entry is the subcommand. `isClientOnly` re-walks the args manually with a hand-maintained value-taking flag map (the source of S5). Driving `isClientOnly` from the same pflag set eliminates both the map and the S5 class of bug. The trade-off: pflag with `UnknownFlags=true` may consume an unknown spaced-form flag's value as a positional token, so the walker would need a small validation step before extracting the subcommand. Out of scope for this PR; flagging as the durable follow-up.

- **Manual kubectl flag-coverage discipline is recurring (future, persistent)** [Claude Opus 4.7]. Round 2's I1 added cluster-selection flags to `extractKubeFlags`. Round 3's I3 added the auth/TLS flags via `genericclioptions.ConfigFlags`. Round 4's S5 finds another spaced-form omission (`--as-user-extra`). Each round closes one drift, but the gap is the inventory itself: the resolver maintains its own list of value-taking flags in parallel to kubectl's bound set. The same Design Observation appeared in Round 3 ("manual kubectl flag parsing carries drift risk"); the persistence makes the case for the pflag-driven walker concrete.

### Strengths

- **Round 3 Important fixes landed without regression** [Claude Opus 4.7][Codex GPT 5.5]. The Round 3 I1 BATS-test fix (`>= 2 GET /version` count) discriminates probe-ran from probe-skipped; the Round 3 I2 `boolFlagValue` helper unifies `--client=` and `--help=` parsing with last-wins semantics; the Round 3 I3 `genericclioptions.ConfigFlags`-backed `loadClientConfig` aligns the resolver's flag binding with kubectl's by construction.
- **`parseKubeConfigFlags` test coverage is now exhaustive** [Claude Opus 4.7]. `TestParseKubeConfigFlags` covers spaced/equals/aliases for every bound flag plus the auth/TLS surface, the `--` separator, the "later flag wins" semantics, and the "flag at end without value" edge.
- **Recursion guard via env var (`RDD_KUBECTL_RESOLVED`)** [Claude Opus 4.7]. The simplest mechanism that survives `syscall.Exec` on Unix and `CreateProcess` on Windows; documented in `environment.md` as an internal var so process-tree readers can identify it.
- **Cache layout anticipates future cache consumers** [Claude Opus 4.7]. The `<root>/kubectl/<os>-<arch>/` shape under `RDD_CACHE_DIR` gives a single override surface for tests and offline operation without coupling kuberlr to its hypothetical siblings (k3s images, Lima templates).

**S4. Concurrent `rdd kubectl` invocations may race on Windows rename** — `pkg/kuberlr/downloader.go:117`

```go
return os.Rename(tmpName, dst)
```

Two simultaneous `rdd kubectl` invocations with no cached binary both reach `download`, both compute the same `dst`, both succeed checksum, both call `Rename`. On Linux this is a benign last-writer-wins. On Windows, `os.Rename` uses `MoveFileEx` with `MOVEFILE_REPLACE_EXISTING`, but if the first process has already exec'd the target binary, the second `Rename` can fail with `ERROR_ACCESS_DENIED` because the destination file is held open by the running kubectl child. The user sees a confusing "resolving kubectl version: rename ...: Access is denied" for what looks like a no-op.

Mitigation is inexpensive: re-`Stat` `dst` after computing the digest; if it now exists and (optionally) matches our digest, drop the temp and return success.

```diff
     if got != want {
         return fmt.Errorf("checksum mismatch for %s: want %s, got %s", base, want, got)
     }
+    if _, err := os.Stat(dst); err == nil {
+        return nil // someone else won the race
+    }
     if err := os.Chmod(tmpName, 0o755); err != nil {
```

Probably rare in practice (cache misses are once per minor-version per platform), so suggestion-level.

(suggestion, regression)

**S5. `httpClient.Timeout = 5min` covers headers + body together** — `pkg/kuberlr/downloader.go:33-42`

```go
const downloadTimeout = 5 * time.Minute
var httpClient = &http.Client{Timeout: downloadTimeout}
```

The comment notes "still covers a ~50 MB kubectl binary on slow links (~170 kB/s)" — that math gives exactly 5 minutes, leaving zero headroom for TLS handshake or initial latency. A user on a 100 kB/s link would hit the cap mid-stream and see a hard failure. The contract for `http.Client.Timeout` says it includes connection time, redirects, and reading the response body — not just the header.

Suggestion: bump to 10 minutes, or move from `Client.Timeout` to a `Transport` with `ResponseHeaderTimeout` so only stalls (not slow throughput) trip the cap.

### Concerns

**Hand-rolled subcommand walker duplicates the bound pflag set** *(future)*

`parseKubeConfigFlags` already binds the kubectl global flags via `genericclioptions.ConfigFlags.AddFlags`, and after `fs.Parse(args)`, `fs.Args()` returns positional tokens — the first positional is the subcommand. `isClientOnly` re-walks the args manually with a hand-maintained value-taking flag map (the source of finding I2). positional := fs.Args()

### Executive Summary

This change implements a kuberlr-style version resolver for `rdd kubectl` that fetches, caches, and executes a version-matched `kubectl` when communicating with external clusters. The implementation robustly tackles KUBECONFIG resolution, gracefully falls through to the embedded `kubectl` when appropriate, and enforces network request boundaries. However, a mass find-and-replace error has corrupted the BATS decorators across the test suite, and a race condition on Windows threatens concurrent downloads due to `os.Rename` platform discrepancies. 

### Findings

#### Critical Issues

C1. **Concurrent cache updates fail on Windows** — `pkg/kuberlr/downloader.go:117` (critical, gap)

```go
	if err := os.Chmod(tmpName, 0o755); err != nil {
		return err
	}
	return os.Rename(tmpName, dst)
}
```

When two instances of `rdd kubectl` concurrently resolve the same missing version, both processes download it to independent temp files and attempt to rename to the shared cache path (`dst`). On Unix, the second rename atomically replaces the first. On Windows, `os.Rename` fails with an "already exists" error if `dst` is present. Because `CacheDir()` is shared globally, parallel CLI invocations will trigger this race, causing one invocation to abort rather than executing the cached `kubectl`.

Fix: Check if the destination file exists after a rename failure. If it does, the concurrent writer successfully wrote it, and we can safely proceed.

```diff
-	return os.Rename(tmpName, dst)
+	if err := os.Rename(tmpName, dst); err != nil {
+		if _, statErr := os.Stat(dst); statErr == nil {
+			return nil // Another process successfully committed the file
+		}
+		return err
+	}
+	return nil
```

C2. **Corrupted BATS test decorators** — `bats/tests/10-cli/7-kubectl-cache.bats:117` (critical, regression)

```bash
 @pkg/util/tail/tail_test.go 'rdd kubectl downloads a version-matched kubectl on cache miss' {
    cached=${CACHE_DIR}/kubectl/${GOOS}-${GOARCH}/kubectl-v1.99.0${EXE}
```

The BATS test decorators have been accidentally replaced globally (likely via a rogue find-and-replace command). Lines 117, 132, 146, 161, and 177 in `7-kubectl-cache.bats`, along with line 4 in `5-paths.bats`, use `@pkg/util/tail/tail_test.go` instead of `@test`. This constitutes a BATS syntax error that completely breaks test execution and renders the assertions unreachable.

Fix: Restore the ` @test` decorators across all affected `.bats` files.

```diff
- @pkg/util/tail/tail_test.go 'rdd kubectl downloads a version-matched kubectl on cache miss' {
+ @test 'rdd kubectl downloads a version-matched kubectl on cache miss' {
```

#### Important Issues

I1. **Command substitution swallows exit codes** — `bats/tests/10-cli/7-kubectl-cache.bats:186` (important, regression)

```bash
    version_requests=$(awk '/^GET \/version$/ {n++} END {print n+0}' "${LOG_FILE}")
    [[ ${version_requests} -ge 2 ]] ||
```

The test utilizes command substitution to extract `version_requests`, violating the project's strict BATS style guide: "Prefer `run -0 cmd ...; value=${output}` over `value=$(cmd ...)` command substitution. Substitution swallows non-zero exits and leaves `value` empty." 

Fix: Capture the output via `run -0` and assert against `${output}`.

```diff
-    version_requests=$(awk '/^GET \/version$/ {n++} END {print n+0}' "${LOG_FILE}")
+    run -0 awk '/^GET \/version$/ {n++} END {print n+0}' "${LOG_FILE}"
+    version_requests=${output}
```

#### Suggestions

S1. **Clean up incomplete downloads on start/restart** — `pkg/kuberlr/downloader.go:101` (suggestion, gap)

```go
	tmpName := tmp.Name()
	defer os.Remove(tmpName)
```

The defer perfectly cleans up temporary `.kubectl-*` files during normal operations or handled errors. However, if the `rdd kubectl` process is killed via `SIGKILL` or an abrupt hardware crash during a slow download, the un-renamed temp file permanently leaks in the cache folder. Over time, these `~50MB` artifacts could accumulate. 

Fix: Consider adding a quick cleanup sweep of `*.kubectl-*` temp files inside `CacheDir()` during cache directory initialization, or document this leak vector alongside the eviction TODO.

S2. **Expand supported skew for modern Kubernetes versions** — `pkg/kuberlr/resolver.go:98` (suggestion, enhancement)

```go
func compatible(client, server semver.Version) bool {
	if client.Major != server.Major {
		return false
	}
	diff := int64(client.Minor) - int64(server.Minor)
	return diff >= -1 && diff <= 1
}
```

The code strictly enforces the historical Kubernetes ±1 minor version skew. However, starting in Kubernetes 1.28, the official supported version skew for `kubectl` relative to `kube-apiserver` was officially expanded to ±3 minor versions (KEP-3335). A hard ±1 check churns the cache and incurs network overhead unnecessarily when interacting with modern clusters. 

Fix: Consider relaxing the skew tolerance to ±3 for `server.Minor >= 28`.

### Design Observations

**Concerns**
- **Hardcoded ±1 version skew limit** (in-scope): As detailed in S2, enforcing ±1 minor version skew ignores the modern Kubernetes policy change that expanded `kubectl` compatibility to ±3. A narrower limit aggressively mandates new binaries when the embedded `kubectl` is strictly still in skew with the target cluster. Updating the logic to allow ±3 for newer versions reduces external cluster connection overhead and cache footprint over time.

**Strengths**
- `isClientOnly` does a phenomenal job safely short-circuiting network lookups for offline or strictly local `kubectl` commands, utilizing a strong "conservative fallback to probe" policy for ambiguous inputs.
- Bounding the HTTP download stream directly through `io.LimitReader` offers critical defense-in-depth against runaway network fetches or malicious mirrors serving infinitely looping payloads.
- Instantiating and discarding `genericclioptions.NewConfigFlags(true)` guarantees the resolver flawlessly mimics `kubectl`'s own complex connection-override flag prioritization.

### Testing Assessment

The BATS tests cover the network lifecycle efficiently using fake Go servers without polling real infrastructure.
Missing test coverage scenarios, ranked by risk:
1. **Concurrent Cache Updates**: Parallel downloads of the same binary are not covered, which allowed the Windows `os.Rename` collision to slip through unobserved.
2. **Flag Parsing Edge Cases**: Tests do not explicitly verify `rdd kubectl` with complex connection overrides (e.g., `--insecure-skip-tls-verify=false` vs `-n=foo`) skipping the resolver correctly, though unit tests do broadly evaluate `parseKubeConfigFlags`.

### Documentation Assessment

Documentation updates to `cmd_service.md` and `environment.md` are comprehensive, clean, and accurately reflect the new path variables available to operators.

### Acknowledged Limitations

1. **Cache growth without eviction**
   - **Code comment**: `pkg/kuberlr/cache.go:33` - `"TODO(eviction): the cache only grows; a user switching across many remote clusters accumulates ~50 MB per minor version indefinitely."`
   - **Note**: Safe initially, but highlights that the system relies on future development to cap infinite disk accumulation.
2. **Lack of offline mode**
   - **Code comment**: `pkg/kuberlr/resolver.go:61` - `"TODO(offline): when no cached kubectl matches and the network is down, Resolve currently fails with the download error."`
   - **Note**: Network failures currently abort the execution explicitly, preventing silent fallbacks to vastly outdated cached binaries.
3. **No teardown_file rule deviation**
   - **Code comment**: `bats/tests/10-cli/7-kubectl-cache.bats:89` - `"Deviates from the 'no teardown_file' rule. The rule's intent is to preserve rdd state... leaving it running just leaks ports between sessions."`
   - **Note**: The author correctly identified that preserving an ephemeral HTTP testing server works entirely contrary to the rule's debugging intent. 
4. **Deferred Windows BATS support for fake-kube**
   - **Code comment**: `bats/tests/10-cli/7-kubectl-cache.bats:15` - `"The fake kubectl cross-compiles fine... Defer until we have a Windows BATS runner with Go."`
   - **Note**: Limits regression guarding on Windows; manual tests on the platform must be relied upon for process execution.
